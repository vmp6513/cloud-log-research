## 1. 设计背景与目标

### 1.1 为什么需要归一化？

本次接入的三个平台——**AWS CloudTrail**、**Azure Activity Log / Resource Logs**、**Microsoft Entra ID**——在日志语义、字段命名、身份表达方式上存在根本性差异：

| 维度 | AWS CloudTrail | Azure Activity Log | Entra ID |
|---|---|---|---|
| 操作主体字段 | `userIdentity`（嵌套对象，含 type/arn/accountId） | `caller`（UPN 或服务身份）+ `identity` 对象 | `initiatedBy`（user/app 二选一） |
| 操作名称字段 | `eventName`（如 `AttachRolePolicy`） | `operationName`（如 `Microsoft.Authorization/roleAssignments/write`） | `activityDisplayName`（如 `Add member to role`） |
| 操作结果字段 | `errorCode` / `errorMessage`（成功时为空） | `resultType`（Success/Failure）+ `resultDescription` | `result`（success/failure） |
| 目标资源字段 | `resources[]` / `requestParameters`（非结构化，API 各异） | `resourceId`（ARM 路径） | `targetResources[]`（对象数组） |
| 时间字段 | `eventTime`（ISO 8601 UTC） | `eventTimestamp`（ISO 8601，含时区偏移） | `activityDateTime`（ISO 8601 UTC） |
| 身份类型区分 | `userIdentity.type`（IAMUser / AssumedRole / AWSService / Root 等） | `identity.claims["...objectidentifier"]` | `initiatedBy.user.id` 或 `initiatedBy.app.appId` |

如果上层研判引擎和 LLM 直接消费原始日志，需要为每个平台各自维护一套字段解析逻辑，规则维护成本极高，跨平台攻击链关联几乎无法实现。归一化的核心价值是：**将平台差异下沉到采集层解决，让研判层和 LLM 只面对一种语言。**

### 1.2 归一化设计目标

- **信息完备性**：每条归一化日志必须能回答"谁 · 在什么时间 · 从哪里 · 对什么 · 做了什么 · 结果如何"
- **研判可用性**：字段语义确定，支持规则型告警（精确匹配）、关联分析（主键关联）、行为统计（数值字段）
- **LLM 友好性**：提供 `action.summary` 自然语言摘要字段，LLM 无需解析嵌套对象即可理解事件语义
- **跨平台关联**：统一 `actor` / `target` / `correlation_id` 等关联键，支持跨平台攻击链还原
- **可扩展性**：平台专有字段通过 `raw_detail` / `enrichment` 扩展字段保留，不强制丢弃

## 2. 建模方法论：为什么这样设计

### 2.1 参考框架：OCSF（裁剪使用）

OCSF（Open Cybersecurity Schema Framework）是由 AWS、Splunk、CrowdStrike 等 18 家厂商于 2022 年联合发布的开源安全日志标准，其最大优势是**把安全事件抽象为"活动（Activity）+ 类别（Class）+ 行为者（Actor）+ 客体（Object）+ 结果（Status）"的五元组结构**，与本方案的需求（谁·时间·对象·操作·结果）高度吻合。

但 OCSF 的完整实现字段集超过 200 个，对于面向实时事中告警的场景过于重。因此本方案**以 OCSF 的语义框架为参考，裁剪为最小可用集，并针对云控制面 / 身份平面的特性补充专属字段**。

### 2.2 核心建模决策

#### 决策一：以"事件"为基本单元，不聚合

每条原始日志对应一条归一化事件，不在归一化层做聚合（聚合由研判层完成）。原因：LLM 研判需要看到独立的上下文；规则型告警基于单条或有限关联；聚合会不可逆地丢失字段信息。

#### 决策二：Actor 统一为主体对象（而非仅 userID 字符串）

三个平台对"谁操作"的表达差异最大：AWS 有 IAMUser / AssumedRole / AWSService / Root 四种身份类型；Azure 有 User / ServicePrincipal / ManagedIdentity；Entra 有 User / Application。**归一化后统一抽象为 `actor` 对象，强制区分 `actor.type`（human / service_account / machine_identity / service）**，这是攻击检测中最关键的语义——Managed Identity 或 AssumedRole 发起的高危操作，与人类用户操作需要触发不同的告警规则。

#### 决策三：Action 拆分为 verb + service + resource_type 三元组

AWS 的 `eventName` 如 `AttachRolePolicy` 是驼峰命名的动词+名词；Azure 的 `operationName` 是 `Microsoft.Authorization/roleAssignments/write` 这样的命名空间路径；Entra 是自然语言 `Add member to role`。**归一化后统一提取 `action.verb`（create/read/update/delete/auth/assume 等）、`action.service`（云服务名）、`action.resource_type`（操作的资源类型）**，研判规则可以基于 verb 做跨平台统一检测（如：所有平台的 IAM 类 create/update 操作）。

#### 决策四：保留 session_chain 和 correlation_id 用于攻击链还原

根据调研报告，三个平台都存在"服务代调"问题（AWS CloudFormation 代调 IAM、Azure ARM Policy 代调、Entra Graph 代调目录操作），导致操作的真实发起者被间接化。**归一化模型中强制提取并保留 `actor.session_context.session_chain`（AssumeRole 链 / 令牌传递链）和 `actor.session_context.on_behalf_of`**，攻击链还原引擎可以通过这两个字段向上追溯真实的人类操作者。

#### 决策五：富化层独立，不污染归一化核心字段

富化（如 IP 地理位置、威胁情报标签、用户历史基线）是研判所需但不属于原始日志的信息。**富化字段统一放入 `enrichment` 子对象，与原始归一化字段严格区分**，避免富化数据质量问题污染核心字段，也方便在富化服务不可用时降级处理。

#### 决策六：action.summary 字段专为 LLM 设计

研判 LLM 接收的不应该是裸 JSON，而是已经人类可读的事件摘要。**归一化层自动生成 `action.summary` 字段**，模板为：`"{actor.display_name}（{actor.type}）通过 {source.ip} 对 [{target.display_name}] 执行了 {action.verb}:{action.resource_type}[{action.name}]，结果：{outcome.result}"`。这让 LLM 的告警生成 Prompt 更简洁，同时降低 LLM 对字段路径理解的依赖。

## 3. 归一化日志模型

以下为完整的归一化事件模型定义。顶层为扁平化的核心字段集，部分语义相关字段组织为子对象以保持结构清晰。所有字段名使用 **snake_case**，时间统一为 **Unix 毫秒时间戳（int64）**，同时保留 ISO 8601 UTC 字符串副本以便可读。

### 3.1 元信息层（Meta）

描述该条归一化事件本身的来源和处理信息，供采集管道追踪使用。

| 字段名 | 类型 | 是否必填 | 说明 |
|---|---|---|---|
| `meta.schema_version` | string | 必填 | 模型版本号，如 `"1.0.0"`，用于下游解析器的 Schema 演进兼容 |
| `meta.event_id` | string | 必填 | 全局唯一事件 ID，UUID v4 格式，由归一化管道生成，非原始日志 ID |
| `meta.source_platform` | string | 必填 | 来源平台枚举：`aws` \| `azure` \| `entra_id` |
| `meta.source_log_type` | string | 必填 | 来源日志类型：`cloudtrail_mgmt` \| `cloudtrail_data` \| `azure_activity` \| `azure_resource` \| `entra_signin` \| `entra_audit` \| `entra_graph` |
| `meta.raw_event_id` | string | 必填 | 原始日志自带的事件 ID（CloudTrail `eventID` / Azure `correlationId` / Entra `id`） |
| `meta.ingested_at` | int64 | 必填 | 归一化管道处理时间（Unix 毫秒），用于计算采集延迟 |
| `meta.account_id` | string | 必填 | AWS Account ID / Azure Subscription ID / Entra Tenant ID，跨账号关联的基础键 |
| `meta.region` | string | 选填 | 事件发生的 Region（AWS / Azure）；Entra 事件填 `"global"` |

### 3.2 时间层（Time）

| 字段名 | 类型 | 是否必填 | 说明 |
|---|---|---|---|
| `time.occurred_at` | int64 | 必填 | 事件发生时间，Unix 毫秒，UTC。来源：AWS `eventTime` / Azure `eventTimestamp` / Entra `activityDateTime` |
| `time.occurred_at_iso` | string | 必填 | ISO 8601 UTC 字符串，如 `"2026-03-15T08:23:44.123Z"`，LLM 可读 |
| `time.received_at` | int64 | 选填 | 平台将日志交付至采集端的时间（如 CloudTrail S3 写入时间），用于检测日志延迟 |

### 3.3 行为主体层（Actor）

**这是归一化中最复杂的层。** 三个平台的身份表达差异最大，且跨角色扮演（AWS AssumeRole 链、Entra 令牌传递）是攻击链还原的核心难点。

| 字段名 | 类型 | 是否必填 | 说明 |
|---|---|---|---|
| `actor.type` | string | 必填 | 身份类型枚举（统一语义）：`human` \| `service_account` \| `machine_identity` \| `cloud_service`。`human`=真实用户；`service_account`=应用/服务主体/IAM User 用于自动化；`machine_identity`=EC2 Instance Role/Managed Identity/Workload Identity；`cloud_service`=平台自身服务（如 `cloudformation.amazonaws.com`） |
| `actor.id` | string | 必填 | 平台原生身份标识符：AWS IAM User ARN 或 AssumedRole ARN / Azure Object ID / Entra User ID or App ID |
| `actor.display_name` | string | 推荐 | 人类可读的身份名称：AWS Username / Azure UPN 或服务主体名 / Entra UserPrincipalName。LLM 生成告警时优先使用此字段 |
| `actor.account_id` | string | 必填 | 操作者所属账号 ID（与 `meta.account_id` 可能不同，如跨账号 AssumeRole 场景，操作者来自另一账号） |
| `actor.credential_type` | string | 必填 | 凭证类型枚举：`password` \| `access_key` \| `assumed_role` \| `managed_identity` \| `oauth_token` \| `certificate` \| `saml` \| `oidc` |
| `actor.mfa_used` | boolean | 选填 | 操作是否经过 MFA 验证（主要用于 Entra SignInLogs 和 AWS 控制台登录场景） |
| `actor.session_context` | object | 选填 | 角色扮演/令牌继承上下文，见 3.3.1 |

#### 3.3.1 actor.session_context（角色链与令牌传递）

**设计意图**：此字段专门解决调研报告中提到的"角色链混淆溯源"问题。AWS 的 AssumeRole 链（A→B→C→D）和 Entra 的令牌传递（App 代表用户）都需要记录"调用链"才能找到最初的人类操作者。

| 字段名 | 类型 | 是否必填 | 说明 |
|---|---|---|---|
| `actor.session_context.session_id` | string | 选填 | AWS STS Session ID / Entra Token ID，在同一角色会话内的所有操作共享此 ID，用于会话级行为聚合 |
| `actor.session_context.assumed_from` | string | 选填 | 当前身份是从哪个身份扮演而来（AWS：上一层 `sessionIssuer` ARN；Entra：令牌颁发的应用 ID） |
| `actor.session_context.session_chain` | string[] | 选填 | AssumeRole 调用链，从最初发起者到当前执行者的完整 ARN 列表。研判引擎可直接遍历此数组还原攻击路径 |
| `actor.session_context.oidc_subject` | string | 选填 | OIDC 令牌中的 `sub` 字段（如 GitHub Actions 场景的 `"repo:org/repo:ref:refs/heads/main"`），是供应链攻击检测的关键字段 |
| `actor.session_context.on_behalf_of` | string | 选填 | Azure 服务代调场景（如 ARM Policy 自动修复、CloudFormation 代调）中，记录发起本次代调的原始用户或 Stack 的 `correlationId` |

### 3.4 操作层（Action）

将三个平台差异极大的操作名称统一为 **verb + service + resource_type + name** 四元组，支持跨平台的统一规则编写。

| 字段名 | 类型 | 是否必填 | 说明 |
|---|---|---|---|
| `action.name` | string | 必填 | 原始操作名称（不做转换，保留原始）：AWS `eventName`（如 `AttachRolePolicy`）/ Azure `operationName` 末段（如 `roleAssignments/write`）/ Entra `activityDisplayName`（如 `Add member to role`） |
| `action.verb` | string | 必填 | 操作动词枚举（跨平台统一）：`create` \| `read` \| `update` \| `delete` \| `auth` \| `assume` \| `invoke` \| `enable` \| `disable` \| `deny` \| `other`。映射规则：AWS `Get*/List*/Describe*`=read；`Put*/Create*/Run*`=create；`Delete*`=delete；`AssumeRole`=assume；`Attach*/Update*`=update；Entra `Add`=create；`Consent`=update |
| `action.service` | string | 必填 | 操作涉及的云服务名（标准化小写）：`iam` \| `s3` \| `lambda` \| `sts` \| `cloudtrail` \| `kms` \| `ec2` \| `azure_rbac` \| `azure_keyvault` \| `azure_storage` \| `entra_directory` \| `entra_signin` \| `graph_api` |
| `action.resource_type` | string | 必填 | 操作目标的资源类型（标准化）：`role` \| `policy` \| `user` \| `group` \| `bucket` \| `object` \| `function` \| `secret` \| `key` \| `vm` \| `subscription` \| `application` \| `service_principal` \| `token` \| `session` |
| `action.category` | string | 必填 | 操作语义分类枚举：`identity_management` \| `privilege_escalation` \| `credential_access` \| `data_access` \| `config_change` \| `auth_event` \| `log_tampering` \| `resource_operation`。**此字段是规则型告警的核心过滤键** |
| `action.summary` | string | 必填 | 自然语言摘要，LLM 专用。模板：`"${actor.display_name}（${actor.type}）通过 ${source.ip} 对 [${target.display_name}] 执行了 ${action.verb}:${action.resource_type}，操作：${action.name}，结果：${outcome.result}"` |

### 3.5 来源层（Source）

描述操作发起的网络和环境上下文，是行为基线偏离检测的核心字段。

| 字段名 | 类型 | 是否必填 | 说明 |
|---|---|---|---|
| `source.ip` | string | 推荐 | 发起操作的客户端 IP；AWS `sourceIPAddress` / Azure `callerIpAddress` / Entra `ipAddress`。注意：某些服务代调场景值为服务域名（如 `cloudformation.amazonaws.com`） |
| `source.user_agent` | string | 选填 | 客户端 UA；AWS `userAgent` / Azure `httpRequest.userAgent` / Entra `userAgent` |
| `source.geo` | object | 富化 | IP 地理位置（富化层写入，原始日志不含）：`country` / `city` / `asn` / `org` |
| `source.access_method` | string | 选填 | 访问方式枚举：`console` \| `cli` \| `sdk` \| `api` \| `saml` \| `oauth` \| `device_code`。来源：AWS 控制台登录 / Entra `clientAppUsed` |
| `source.device` | object | 选填 | 设备信息（主要来自 Entra SignInLogs）：`device_id` / `os` / `browser` / `is_compliant` / `is_managed` |

### 3.6 目标层（Target）

统一描述操作对象。三个平台的资源表示方式差异很大（AWS ARN / Azure ResourceId / Entra Object ID），归一化后提供统一的 `target.id`（保留原始格式）和 `target.display_name`（人类可读名称）。

| 字段名 | 类型 | 是否必填 | 说明 |
|---|---|---|---|
| `target.id` | string | 推荐 | 目标资源的原始 ID：AWS ARN / Azure `resourceId`（ARM 路径）/ Entra Object ID |
| `target.display_name` | string | 推荐 | 目标资源的人类可读名称（LLM 告警使用）：Bucket 名 / Role 名 / 用户 UPN / 资源组名 |
| `target.type` | string | 必填 | 目标资源类型（与 `action.resource_type` 保持一致，但从资源角度描述） |
| `target.account_id` | string | 选填 | 目标资源所属账号（与 `actor.account_id` 不同时说明跨账号操作） |
| `target.scope` | string | 选填 | 操作影响范围（Azure/Entra 特有）：`tenant` \| `subscription` \| `resource_group` \| `resource`；AWS 场景填 `account` \| `region` \| `resource` |

### 3.7 操作结果层（Outcome）

| 字段名 | 类型 | 是否必填 | 说明 |
|---|---|---|---|
| `outcome.result` | string | 必填 | 结果枚举：`success` \| `failure` \| `unknown`。AWS：`errorCode` 为空=success；Azure：`resultType=Succeeded`；Entra：`result=success` |
| `outcome.error_code` | string | 选填 | 原始错误码：AWS `errorCode`（如 `AccessDenied`）/ Azure `resultType`（如 `Failed`）/ Entra `errorCode`（如 `50126`） |
| `outcome.error_message` | string | 选填 | 原始错误信息（截断至 512 字节，避免噪声） |
| `outcome.http_status` | int | 选填 | HTTP 状态码（主要来自 Entra Graph Activity Logs 和 Azure HTTP 请求日志） |

### 3.8 富化层（Enrichment）

**富化层与归一化核心字段严格分离**，由下游富化服务（IP 情报、用户画像、威胁情报）写入，原始采集阶段可为空。上层研判引擎读取时应判断字段是否存在。

| 字段名 | 类型 | 是否必填 | 说明 |
|---|---|---|---|
| `enrichment.threat_intel` | object[] | 选填 | IP / 域名威胁情报标签：`source`（情报源名称）/ `indicator`（命中的 IoC）/ `confidence`（0-100）/ `tags`（如 `c2`, `scanner`, `tor`） |
| `enrichment.actor_profile` | object | 选填 | 用户历史画像（来自行为基线服务）：`typical_ips[]` / `typical_hours` / `typical_services[]` / `risk_score` |
| `enrichment.alert_tags` | string[] | 选填 | 研判层预置的规则命中标签，如 `["t1078.004_cloud_account", "privilege_escalation"]`，使用 MITRE ATT&CK 技术 ID 编码 |
| `enrichment.related_events` | object[] | 选填 | 关联事件引用（由攻击链关联引擎写入）：`meta.event_id` + `relationship`（如 `preceded_by` / `caused_by`） |
| `enrichment.is_service_agent_call` | boolean | 选填 | 标记是否为云平台服务代调（CloudFormation/ARM Policy/Graph 等），提示研判引擎需向上追溯 `on_behalf_of` 字段 |

### 3.9 原始字段保留（RawDetail）

归一化不应丢失原始信息。**`raw_detail` 字段保留平台特有的、归一化字段无法覆盖的关键内容**，以 JSON 字符串形式存储，供深度调查时参考。

| 字段名 | 类型 | 是否必填 | 说明 |
|---|---|---|---|
| `raw_detail.request_parameters` | string（JSON） | 选填 | AWS `requestParameters` / Azure `requestBody` 中的非敏感操作参数（如被创建角色的名称、被附加策略的 ARN） |
| `raw_detail.response_elements` | string（JSON） | 选填 | AWS `responseElements` / 操作返回的关键资源 ID（如新建角色的 `roleId`） |
| `raw_detail.additional_details` | string（JSON） | 选填 | Entra `additionalDetails` 数组、Azure `Properties` 中的扩展字段 |
| `raw_detail.platform_specific` | string（JSON） | 选填 | 无法映射到任何归一化字段的平台特有字段（如 AWS `vpcEndpointId`、Entra `conditionalAccessStatus`） |

## 4. 平台字段映射规则

### 4.1 AWS CloudTrail → 归一化模型

| 归一化字段 | CloudTrail 原始字段 | 映射说明 / 转换逻辑 |
|---|---|---|
| `meta.source_platform` | （固定值） | `"aws"` |
| `meta.raw_event_id` | `eventID` | CloudTrail 事件的全局唯一 UUID |
| `meta.account_id` | `userIdentity.accountId` 或 `recipientAccountId` | 优先取 `recipientAccountId`（事件真实发生的账号） |
| `time.occurred_at` | `eventTime` | ISO 8601 字符串转换为 Unix 毫秒 |
| `actor.type` | `userIdentity.type` | `IAMUser`→`human`；`AssumedRole`→`service_account` 或 `machine_identity`（视 `sessionIssuer` 判断）；`AWSService`→`cloud_service`；`Root`→`human`；`FederatedUser`→`human` |
| `actor.id` | `userIdentity.arn` | 完整 ARN，唯一标识 |
| `actor.display_name` | `userIdentity.userName` | IAMUser 场景有值；AssumedRole 取 ARN 末段 Session Name |
| `actor.credential_type` | `userIdentity.type` + `sessionContext` | `AssumedRole`→`assumed_role`；`IAMUser`+accessKeyId→`access_key`；控制台登录→`password`；`FederatedUser`→`saml`/`oidc` |
| `actor.session_context.session_chain` | `userIdentity.sessionContext.sessionIssuer`（逐层解析） | 通过 `sessionIssuer` 递归构建调用链数组，最多追溯 10 层 |
| `actor.session_context.oidc_subject` | `responseElements.subjectFromWebIdentityToken` | `AssumeRoleWithWebIdentity` 事件专有，是供应链攻击检测的关键字段 |
| `action.name` | `eventName` | 原样保留，如 `AttachRolePolicy` |
| `action.verb` | `eventName` 前缀解析 | `Get*/List*/Describe*`=`read`；`Create*/Put*/Run*/Start*`=`create`；`Delete*/Remove*`=`delete`；`Update*/Modify*/Attach*/Detach*`=`update`；`AssumeRole*`=`assume`；`Invoke`=`invoke` |
| `action.service` | `eventSource` | `iam.amazonaws.com`→`iam`；`s3.amazonaws.com`→`s3`；`sts.amazonaws.com`→`sts`；`lambda.amazonaws.com`→`lambda`；`cloudtrail.amazonaws.com`→`cloudtrail`；`kms.amazonaws.com`→`kms` |
| `action.category` | `eventSource` + `eventName` 组合 | `iam.*`（AttachRolePolicy/CreateUser）→`privilege_escalation`/`identity_management`；`sts.AssumeRole*`→`auth_event`；`cloudtrail.StopLogging`/`DeleteTrail`→`log_tampering`；`s3.GetObject`（数据事件）→`data_access` |
| `source.ip` | `sourceIPAddress` | 直接映射；若值为服务域名（含 `.amazonaws.com`）则同步置 `enrichment.is_service_agent_call=true` |
| `target.id` | `resources[0].ARN` 或 `requestParameters` 中的资源 ARN | 优先取 `resources[].ARN`；无则从 `requestParameters` 按 API 语义解析 |
| `outcome.result` | `errorCode` | `errorCode` 为空→`success`；`AccessDenied`/其他错误→`failure` |
| `outcome.error_code` | `errorCode` | 直接映射 |

## 6. 日志富化策略

富化在归一化之后、研判之前完成，目的是将归一化日志中缺失但对告警研判有价值的上下文信息补入 `enrichment` 对象。

### 6.1 富化优先级矩阵

| 富化类型 | 写入字段 | 数据来源 | 对研判的价值 |
|---|---|---|---|
| IP 威胁情报 | `enrichment.threat_intel` | 内部 TIP / AlienVault OTX / GreyNoise | 识别来自已知 C2、Tor、扫描器的操作；是触发高优先级告警的最强信号之一 |
| IP 地理位置 | `enrichment.source.geo` | MaxMind GeoIP2 / ip-api | 支持"首次来自新国家/新 ASN"的行为偏离检测 |
| 用户历史画像 | `enrichment.actor_profile` | 行为基线服务（历史 30 天统计） | 支持"非常规时间段操作"、"首次使用此 IP"、"操作量突增"等基线偏离告警；对 LLM 输出"为什么可疑"的解释至关重要 |
| MITRE ATT&CK 标签 | `enrichment.alert_tags` | 归一化层根据 `action.name` + `action.category` 预置标签 | 研判引擎基于 ATT&CK 技术 ID 聚合同一攻击阶段的多个事件，支持攻击链识别 |
| 关联事件引用 | `enrichment.related_events` | 攻击链关联引擎（基于 `correlation_id` / `session_id` / `actor.id` 时间窗关联） | LLM 可以在一次 Prompt 中看到整条攻击链（前置侦察→提权→横移），告警质量和上下文完整性大幅提升 |
| OAuth 应用白名单 | `enrichment.alert_tags += "unknown_app"` | 租户已知应用白名单（来自 Entra App 注册列表） | Entra AuditLogs 中 `target.id` 不在白名单时自动打标，对 OAuth 钓鱼 / Storm-0558 类攻击是早期检测点 |

### 6.2 针对 LLM 研判的 action.summary 生成规范

`action.summary` 字段在归一化层生成，是 LLM 告警生成 Prompt 的主要输入内容。

```python
# 生成模板（伪代码）

  "{actor_display}（{actor_type_cn}）通过 {source_ip} 对 [{target_display}] 执行了 {verb_cn}:{resource_type_cn}，"
  "操作：{action_name}，结果：{outcome_result_cn}",
  {
    actor_display:     actor.display_name ?? actor.id,
    actor_type_cn:     {"human": "用户", "service_account": "服务账户",
                        "machine_identity": "机器身份", "cloud_service": "云服务"}[actor.type],
    source_ip:         source.ip ?? "内部服务调用",
    target_display:    target.display_name ?? target.id ?? "未知目标",
    verb_cn:           {"create": "创建", "read": "读取", "update": "修改", "delete": "删除",
                        "auth": "认证", "assume": "扮演角色", "invoke": "调用"}[action.verb],
    resource_type_cn:  action.resource_type,
    action_name:       action.name,
    outcome_result_cn: {"success": "成功", "failure": "失败", "unknown": "未知"}[outcome.result]
  }
)

# 输出示例：

#  操作：AttachRolePolicy，结果：成功"
```

## 7. 关键检测场景与字段使用说明

### 7.1 跨平台高危 IAM 操作检测

**检测目标**：任意平台上的权限提升操作（附加高权限策略、添加到特权角色、OAuth 高危授权）

```
ALERT "高危权限变更" WHEN
  action.category = "privilege_escalation"
  AND action.verb IN ("create", "update")
  AND action.service IN ("iam", "azure_rbac", "entra_directory")
  AND outcome.result = "success"

// 附加条件（提高精度）
ESCALATE TO "P1" WHEN
  raw_detail.request_parameters CONTAINS "AdministratorAccess"        // AWS
  OR raw_detail.request_parameters CONTAINS "Owner"                   // Azure
  OR raw_detail.additional_details CONTAINS "RoleManagement.ReadWrite"  // Entra
```

### 7.2 机器身份发起的异常操作检测

**检测目标**：EC2 Instance Role / Managed Identity / Workload Identity 发起的 RBAC 变更（横向移动的早期信号）

```
ALERT "机器身份异常提权" WHEN
  actor.type = "machine_identity"
  AND action.category IN ("privilege_escalation", "identity_management")
  AND outcome.result = "success"

// 联动富化字段
ESCALATE IF enrichment.threat_intel != null
  OR enrichment.actor_profile.typical_services NOT CONTAINS action.service
```

### 7.3 日志清除 / 审计规避操作检测（最高优先级）

**检测目标**：攻击者获得管理员权限后的善后行为，是响应窗口最短的事件

> 覆盖范围：AWS `cloudtrail.StopLogging` / `DeleteTrail`，Azure `microsoft.insights/diagnosticSettings/delete`，Entra `microsoft.aadiam/diagnosticsettings/delete`

```
ALERT "P0: 审计日志操作" WHEN
  action.category = "log_tampering"

// 要求：走实时路径，不走批量扫描
// 目标延迟：从日志产生到告警触发 < 5 分钟
```

### 7.4 角色链攻击溯源（跨事件关联）

**检测目标**：还原 AWS AssumeRole 链或供应链攻击的 OIDC 入口（UNC6426 场景）

```
// 关联查询（基于 actor.session_context）
CHAIN ANALYSIS:
  1. 查找 action.verb="assume" 事件序列（同一 meta.account_id 内）
  2. 通过 actor.session_context.session_chain 数组还原调用路径
  3. 检查链路中是否有 actor.session_context.oidc_subject 不在白名单的节点

// OIDC 白名单告警
ALERT "可疑 OIDC 令牌来源" WHEN
  action.name = "AssumeRoleWithWebIdentity"
  AND actor.session_context.oidc_subject NOT MATCHES "repo:{YOUR_ORG}/*"
```


## 8. 模型局限性与已知盲区

归一化模型覆盖了三个平台的控制面和部分数据面日志，但以下盲区在调研报告中已明确指出，需要在产品能力说明中向用户披露：

| 盲区类型 | 影响的攻击场景 | 建议补充手段 |
|---|---|---|
| EC2/VM 内部进程行为 | Ransomware、内存马、供应链注入后执行 | AWS CloudWatch Agent + GuardDuty Runtime；Azure Monitor Agent + Defender for Endpoint |
| 网络流量内容（仅有元数据） | 横向移动、数据外传 | AWS VPC Flow Logs；Azure NSG Flow Logs。仅五元组，无 Payload |
| AWS SSM Run Command 命令内容 | 攻击者通过 SSM 在 EC2 执行恶意命令 | 配置 SSM 日志流式传输至 CloudWatch；或部署 EDR |
| Entra MicrosoftGraphActivityLogs（默认关闭） | OAuth 令牌被滥用后的数据访问（Storm-0558 类攻击） | 接入时强制检查并开启；通过诊断设置导出到 Event Hub |
| 本地端侧凭证窃取 | QUIETVAULT、键盘记录、内存 dump | EDR（CrowdStrike / SentinelOne）；此类事件在云平台日志中完全不可见 |
| 跨账号 / 跨租户操作的完整链路 | AWS 跨账号 AssumeRole、Azure 跨订阅横移、B2B 跨租户访问 | 必须同时接入所有相关账号/订阅/租户；单一账号接入无法看到跨账号攻击全貌 |
| AWS 数据事件（S3 GetObject 等，默认不开启） | 数据泄露（具体泄露了哪些对象） | 客户需要提前开启 CloudTrail 数据事件（额外付费）；否则泄露后无法追溯被访问的具体对象 |
