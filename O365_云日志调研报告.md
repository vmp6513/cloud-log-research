## 1. 日志分类

M365 的日志体系由两大相对独立的系统构成：**Microsoft Purview 统一审计日志（UAL）** 和 **Microsoft Entra ID 活动日志**，二者在数据内容、API、保留策略上均有差异，切勿混淆。

### 1.1 总览

| 日志类别 | 子类 | 来源 API | 简要描述与安全意义 |
|---|---|---|---|
| **Purview 统一审计日志（UAL）** | Audit (Standard) | Office 365 Management Activity API / Search-UnifiedAuditLog | 覆盖 Exchange、SharePoint、OneDrive、Teams、Entra ID 等 30+ 服务的用户和管理员操作；是 M365 安全溯源的核心数据源 |
| | Audit (Premium) | 同上 | 在 Standard 基础上增加高价值事件（如 `MailItemsAccessed`）、更长保留和更高 API 带宽 |
| **Microsoft Entra ID 活动日志** | 登录日志（Sign-in logs） | Microsoft Graph API (`/auditLogs/signIns`) | 记录交互式、非交互式、服务主体、托管标识的登录事件；是身份攻击检测的核心 |
| | 审计日志（Audit logs） | Microsoft Graph API (`/auditLogs/directoryAudits`) | 记录 Entra ID 目录层面的变更操作，如角色分配、用户创建、条件访问策略变更 |
| | 预配日志（Provisioning logs） | Microsoft Graph API (`/auditLogs/provisioning`) | 记录 Entra ID 用户预配（同步）的结果，适用于混合环境中的身份治理 |
| | Microsoft Graph 活动日志 | 诊断设置 → Log Analytics / Event Hub | 记录所有对租户 Microsoft Graph API 的 HTTP 请求；是检测令牌滥用（如 OAuth 攻击）的重要数据源（**需 P1/P2 + 诊断设置配置才可获取**） |

### 1.2 UAL 核心内容类型（Content Types）

Office 365 Management Activity API 将 UAL 按工作负载聚合为 5 类内容类型：

| Content Type | 覆盖范围 |
|---|---|
| `Audit.AzureActiveDirectory` | Entra ID 目录操作、用户登录（含 `UserLoggedIn`） |
| `Audit.Exchange` | Exchange 邮箱操作（读取、删除、规则变更）及 Exchange 管理员操作 |
| `Audit.SharePoint` | SharePoint / OneDrive 文件操作（访问、下载、分享、删除） |
| `Audit.General` | Teams、Power BI、Viva 等其他 M365 服务 |
| `DLP.All` | 数据丢失防护（DLP）策略匹配事件 |

> **参考文档**：  
> - UAL 总览：[Learn about auditing solutions in Microsoft Purview](https://learn.microsoft.com/en-us/purview/audit-solutions-overview)  
> - Content Types 定义：[Office 365 Management Activity API reference](https://learn.microsoft.com/en-us/office/office-365-management-api/office-365-management-activity-api-reference)  
> - Entra ID 日志分类：[Logs available for streaming from Microsoft Entra ID](https://learn.microsoft.com/en-us/entra/identity/monitoring-health/concept-diagnostic-settings-logs-options)

## 2. 能力边界

### 2.1 视角边界

**控制面可见，数据面不可见。**

M365 审计日志的视角停留在**操作行为层**：能看到"谁做了什么动作、针对哪个对象、在什么时间、从哪个 IP"，但无法获得任何网络流量（PCAP/NetFlow）、邮件正文或文件内容本身。

具体边界如下：

| 可见 | 不可见 |
|---|---|
| 用户登录事件（含 IP、客户端、认证方式） | 网络层流量（无 PCAP/NetFlow） |
| 邮件操作（读、删、转发规则） | 邮件正文及附件内容 |
| 文件操作（访问、下载、分享） | 被下载文件的副本/内容 |
| 管理员权限变更（角色、MFA 策略） | 终端进程、内存信息（需 EDR） |
| OAuth 应用授权/撤销 | 网络内部横向移动的流量细节 |
| Teams 消息发送/删除操作 | Teams 消息内容本体 |

**重要限制**：Entra ID 登录日志和 UAL 是**两套独立系统**，Purview 合规搜索无法查询到 Entra 登录日志，需分别采集。

> **参考文档**：[Microsoft Entra data retention](https://learn.microsoft.com/en-us/entra/identity/monitoring-health/reference-reports-data-retention)（明确说明 Entra 日志与 UAL 独立）

---

### 2.2 内容边界

**只记录动作，不记录载体。**

- **文件操作**：记录 `FileDownloaded`（含文件名、路径、操作者 IP），但**不保存被下载文件的内容副本**。
- **邮件访问**：E5 License 下的 `MailItemsAccessed` 记录邮件被访问的事实（包含邮件 ID、客户端类型），但**不记录邮件正文**。
- **搜索行为**：Audit (Premium) 的 `SearchQueryInitiatedExchange` / `SearchQueryInitiatedSharePoint` 记录用户搜索的关键词，但**不记录搜索结果集**。
- **Teams 消息**：记录"消息被删除"的操作事件，但**不记录被删除消息的内容**。

---

### 2.3 日志完整性与可信度（补充项 A）

这是安全厂商经常忽视、但在溯源分析中至关重要的问题。

#### 攻击者能否删除/篡改审计日志？

**结论：攻击者无法直接删除 UAL 中的已有日志记录，但可以关闭未来的日志记录。**

| 操作 | 是否可行 | 说明 |
|---|---|---|
| 关闭 UAL 记录（`Set-AdminAuditLogConfig`） | **可以**（需 Organization Management 角色） | 关闭行为本身会被记录在 Exchange Admin Audit Log 中，但若攻击者同时控制日志查询权限，可规避检测 |
| 删除历史 UAL 记录 | **不可以**（微软未提供此接口） | UAL 历史日志由微软托管，租户管理员无删除能力 |
| 修改/篡改历史 UAL 记录 | **不可以** | 同上 |
| 关闭特定用户的 Mailbox Audit | **可以**（Exchange Online 管理员） | 执行 `Set-Mailbox -AuditEnabled $false` 可停止该用户邮箱操作的记录 |

**安全意义**：
- 如果攻击者获得了 `Organization Management` 或 `Exchange Administrator` 权限，他们**可以静默关闭日志采集**，使后续操作留下更少痕迹。
- 这一行为本身会产生审计记录（`Set-AdminAuditLogConfig`），但若 SIEM 未对此类高危操作设置告警，可能被忽略。
- **最佳实践**：安全厂商应将"UAL 开关状态变更"作为高优先级告警规则，并定期将 UAL 导出至外部不可变存储（如 Azure Storage immutable blob），避免对微软平台的单点依赖。

> **参考文档**：[Turn auditing on or off](https://learn.microsoft.com/en-us/purview/audit-log-enable-disable)  
> **攻防场景参考**：[Advanced Threat Detection in Microsoft 365 – CoreView](https://www.coreview.com/blog/advanced-threat-detection-in-m365)（描述了攻击者关闭 UAL 的 TTP）

### 2.4 日志保留策略与取证窗口（补充项 D）

日志保留时长直接决定了**事后取证能追溯多远**，对 APT 类攻击（潜伏期常超过 90 天）尤为关键。

#### UAL（Purview 审计日志）保留时长

| License 等级 | UAL 默认保留时长 | 可扩展上限 |
|---|---|---|
| Microsoft 365 Business Basic/Standard/Premium | 180 天 | 无（不支持自定义保留策略） |
| Office 365 E1 / Microsoft 365 E3 | 180 天 | 可通过审计保留策略延长（最长 1 年） |
| Microsoft 365 E5 / Audit (Premium) | **1 年**（Entra ID、Exchange、SharePoint 默认） | 可通过附加 License 延长至 **10 年** |

> **注意**：2023 年 10 月 17 日起，Audit (Standard) 默认保留期从 90 天延长至 180 天。2023 年 10 月 17 日之前产生的日志仍按 90 天保留。

#### Entra ID 活动日志保留时长

| License 等级 | 登录日志 / 审计日志 | 预配日志 |
|---|---|---|
| Entra ID Free | 7 天 | 7 天 |
| Entra ID P1 / P2 | **30 天** | 30 天 |
| 通过诊断设置导出 | 由目标存储（Log Analytics / Storage Account）决定，理论上无限 | 同左 |

**高危场景**：Entra ID 登录日志仅保留 30 天（P1/P2），如果未提前配置诊断设置导出，**30 天之前的登录痕迹将永久丢失**，且**无法追溯获取**。这在 BEC 溯源和 APT 追踪中是重大盲点。

> **参考文档**：
> - UAL 保留：[Search the audit log – Retention periods](https://learn.microsoft.com/en-us/purview/audit-search)  
> - Entra 保留：[Microsoft Entra data retention](https://learn.microsoft.com/en-us/entra/identity/monitoring-health/reference-reports-data-retention)  
> - Entra 日志不可追溯说明：[Retrieving sign-in logs beyond 30 days in Entra](https://learn.microsoft.com/en-us/answers/questions/5744078/retrieving-sign-in-logs-or-similar-evidence-beyond)（Q&A 确认：过期 Entra 日志无法找回）

### 2.5 其他限制

以下是容易引发误解的关键限制，按类别梳理：

**① License 决定可见的事件类型，而非仅影响保留时长**

这是最常见的误区。`MailItemsAccessed`（精确记录邮件被哪个应用、哪个 IP 访问）是 Audit (Premium) 专有事件，E3 及以下用户**根本不产生此事件**，而非仅仅是"保留期更短"。同理，`SearchQueryInitiatedExchange` 也仅限 Premium。

| 事件 | Standard | Premium |
|---|---|---|
| `MailItemsAccessed` | ❌ 不产生 | ✅ |
| `Send`（精确邮件发送记录） | ❌ 不产生 | ✅ |
| `SearchQueryInitiatedExchange` | ❌ 不产生 | ✅ |
| `FileDownloaded`、`SharingSet` 等通用事件 | ✅ | ✅ |

**② SMB 授权（Business 系列）默认不开启审计**

Business Basic/Standard/Premium 等 SMB License 的 UAL **默认关闭**，需手动在 Purview 门户开启，且不支持高级保留策略。

**③ UAL 内容类型不支持精确时间过滤**

Management Activity API 的 `/content` 接口按**"内容可用时间"**（即日志被写入 blob 的时间）过滤，**不等于事件发生时间**。这意味着你无法通过 API 精确查询"某个时间段内发生的所有事件"，可能产生漏拉或重复拉取。

**④ SharePoint "Open in App" 功能不产生 `FileDownloaded` 事件**

2024 年，Varonis Threat Labs 发现当用户通过 SharePoint 的"在应用中打开"功能打开文件时，**不会产生 `FileDownloaded` 事件**，这为数据泄露检测留下了盲点。

> **参考文档**：[Varonis Threat Labs – SharePoint Open in App blind spot (2024)](https://www.coreview.com/blog/advanced-threat-detection-in-m365)

**⑤ Entra 日志与 UAL 完全独立，无法互相补充**

Purview 合规搜索（UAL 入口）**不包含** Entra 登录日志，反之亦然。安全厂商若只拉取 UAL，将完全看不到 Entra 层面的登录异常（如异地登录、风险登录）。

## 3. 日志采集流程

M365 提供两套主要的程序化采集方案，分别针对不同的数据源：

| 方案 | 数据源 | 推荐场景 |
|---|---|---|
| **方案 A**：Office 365 Management Activity API | Purview UAL（所有工作负载） | 大规模安全监控、SIEM 接入 |
| **方案 B**：Microsoft Graph API | Entra ID 登录/审计/预配日志 | 身份安全、IAM 集成 |
| **方案 C**：诊断设置（Diagnostic Settings）| Entra ID 日志 → Azure Event Hub / Log Analytics | 实时流式传输、SIEM 集成 |

### 3.1 方案 A：Office 365 Management Activity API（UAL 采集）

#### 3.1.1 概要流程

```
1. 在 Entra ID（Azure AD）中注册应用
2. 为应用授予 ActivityFeed.Read 权限（Application 权限）
3. 管理员授予租户级 Admin Consent
4. 调用 /start 接口订阅所需 Content Type（如 Audit.Exchange）
5. 定期调用 /content 接口轮询可用 Content Blob 列表
6. 逐一下载 Blob URL 中的 JSON 日志数据
7. 解析 JSON，写入 SIEM / 数据仓库
```

> 注：微软推荐使用此 API 而非 PowerShell 脚本进行大规模程序化采集。

#### 3.1.2 核心 API

| API 操作 | 端点示例 | 说明 |
|---|---|---|
| 订阅内容类型 | `POST /api/v1.0/{tenantId}/activity/feed/subscriptions/start?contentType=Audit.Exchange` | 开始订阅指定工作负载的日志 |
| 列出可用内容 | `GET /api/v1.0/{tenantId}/activity/feed/subscriptions/content?contentType=Audit.Exchange&startTime=...&endTime=...` | 返回 Content Blob 的 URL 列表 |
| 下载 Blob | `GET {contentUri}` | 下载实际日志 JSON，单个 Blob 可包含多条事件 |
| 停止订阅 | `POST /api/v1.0/{tenantId}/activity/feed/subscriptions/stop?contentType=Audit.Exchange` | |
| 注册 Webhook | 在 /start 接口中携带 `webhook` 参数 | 推送模式：有新 Blob 时微软主动通知，**不推荐用于生产**（见 §3.1.6） |

基础 URL：`https://manage.office.com`

#### 3.1.3 返回格式

以 JSON 格式返回，每个 Blob 包含**多条**日志记录的 JSON 数组。每条记录包含 `Common Schema`（通用字段）+ `Service-Specific Schema`（工作负载专属字段）。

```json
[
  {
    "CreationTime": "2024-03-15T10:22:31",
    "Id": "a1b2c3d4-...",
    "Operation": "FileDownloaded",
    "OrganizationId": "xxxxx-tenant-id-xxxxx",
    "RecordType": 6,
    "ResultStatus": "Succeeded",
    "UserKey": "user@contoso.com",
    "UserType": 0,
    "Version": 1,
    "Workload": "SharePoint",
    "ClientIP": "203.0.113.45",
    "UserId": "user@contoso.com",
    "ObjectId": "https://contoso.sharepoint.com/sites/.../document.xlsx",
    "SiteUrl": "https://contoso.sharepoint.com/sites/Finance",
    "SourceFileName": "Q4_Finance_Report.xlsx"
  },
  { ... }
]
```

#### 3.1.4 关键字段

| 字段 | 类型 | 说明 |
|---|---|---|
| `CreationTime` | string (UTC) | **事件发生时间**（非日志写入时间），告警研判的主要时间基准 |
| `Id` | GUID | 日志记录唯一标识，用于去重 |
| `Operation` | string | 操作类型（如 `FileDownloaded`、`New-InboxRule`、`UserLoggedIn`），**最重要的分类字段** |
| `Workload` | string | 所属服务（`Exchange`、`SharePoint`、`AzureActiveDirectory` 等） |
| `RecordType` | integer | 日志类型枚举值，与 Workload 对应，用于精确过滤 |
| `UserId` | string | 执行操作的用户 UPN（如 `user@contoso.com`），部分系统操作为服务账号 |
| `UserKey` | string | 用户在 Entra ID 中的唯一标识符（有时与 UserId 相同，但系统事件可能不同） |
| `ClientIP` | string | 操作来源 IP；**注意**：不同 Operation 类型 IP 字段名可能不同（`ClientIP`、`ClientIPAddress`、`ActorIPAddress`），需按类型解析 |
| `ResultStatus` | string | `Succeeded` / `Failed`，部分场景下缺失 |
| `ObjectId` | string | 操作对象（文件 URL、邮件 ID、用户 GUID 等） |
| `AuditData` | JSON string | 包含所有工作负载专属扩展字段，**实际分析必须解析此字段** |

> **IP 字段陷阱**：根据安全研究（[SecurityAura, 2024](https://medium.com/@securityaura/microsoft-unified-audit-log-ual-what-you-come-to-learn-the-hard-way-905a087a1558)），M365 UAL 中 IP 地址在不同 Operation 类型下可能分散在 `ClientIP`、`ClientIPAddress`、`ActorIPAddress` 等多个字段，字段名不统一，需在解析器中做兼容处理。

> **参考文档**：[Office 365 Management Activity API schema](https://learn.microsoft.com/en-us/office/office-365-management-api/office-365-management-activity-api-schema)  
> [Detailed properties in the audit log](https://learn.microsoft.com/en-us/purview/audit-log-detailed-properties)

#### 3.1.5 所需权限

| 权限 | 类型 | 用途 |
|---|---|---|
| `ActivityFeed.Read` | Application（应用权限） | 读取所有内容类型的 UAL 日志（必需） |
| `ActivityFeed.ReadDlp` | Application | 额外读取 DLP 敏感数据内容（可选，需合规审批） |
| `ServiceHealth.Read` | Application | 读取服务健康状态（可选） |

申请步骤：  
1. Entra ID 注册应用（App Registration）  
2. 在"API 权限"中添加 `Office 365 Management APIs` → `ActivityFeed.Read`  
3. 由租户管理员执行 Admin Consent  

> **参考文档**：[Get started with Office 365 Management APIs](https://learn.microsoft.com/en-us/office/office-365-management-api/get-started-with-office-365-management-apis)

#### 3.1.6 安全厂商使用策略与踩坑经验

**推荐采集模式：轮询（Pull），而非 Webhook（Push）**

微软自身在故障排查文档中明确指出：Webhook 模式近期存在**通知延迟**，直接轮询 `/content` 接口比 Webhook 更可靠。生产环境建议使用带状态管理的轮询循环。

**时间窗口设计**：

由于核心服务（Exchange、SharePoint、OneDrive、Teams）的事件从发生到 Blob 可用，典型延迟为 **60-90 分钟**，设计采集时间窗口时应预留此缓冲。建议轮询间隔不超过 5 分钟（用于及时发现新 Blob），但回查窗口至少覆盖过去 2-3 小时以防漏拉。

**分页处理**：

对于大型租户，单次 `/content` 响应可能超出单页容量，需检查响应 Header 中的 `NextPageUri` 并循环翻页，直至无此 Header。

**限流（Throttling）处理**：

- 初始额度：所有租户默认 **2,000 请求/分钟**  
- E5/Audit Premium 租户动态提升，约为 Standard 的两倍  
- 遭遇 HTTP 429 时必须实现指数退避（Exponential Backoff）策略

**Blob 去重**：

Blob 包含的事件并不保证按发生时间排序。如果采集中断后重启，可能重复拉取部分 Blob。建议用 `Id`（记录 GUID）做幂等去重。

> **参考文档**：  
> - [Troubleshooting the Management Activity API](https://learn.microsoft.com/en-us/office/office-365-management-api/troubleshooting-the-office-365-management-activity-api)（官方故障排查，含延迟、Webhook 说明）  
> - [SecurityAura – UAL Hard Lessons (2024)](https://medium.com/@securityaura/microsoft-unified-audit-log-ual-what-you-come-to-learn-the-hard-way-905a087a1558)（从 BEC 调查视角的实战踩坑）

### 3.2 方案 B：Microsoft Graph API（Entra ID 日志采集）

#### 3.2.1 概要流程

```
1. 注册 Entra ID 应用，获取 Application 权限
2. 使用 Client Credentials 流获取 access_token
3. 调用 Graph API 分页拉取登录 / 审计 / 预配日志
4. 使用 $filter 参数进行服务端过滤（按时间、用户等）
5. 增量拉取：记录上次拉取的最大 createdDateTime，下次从此时间点续拉
```

#### 3.2.2 核心 API

| 日志类型 | API 端点 | 许可要求 |
|---|---|---|
| 登录日志 | `GET /v1.0/auditLogs/signIns` | Entra ID P1 或 P2 |
| 审计日志 | `GET /v1.0/auditLogs/directoryAudits` | 任意付费 License |
| 预配日志 | `GET /v1.0/auditLogs/provisioning` | 任意付费 License |

**注意**：Graph API 并非为大批量数据拉取设计。微软文档明确指出，拉取大量数据可能导致分页和性能问题，**生产环境建议通过诊断设置（方案 C）流式传输**。

#### 3.2.3 所需权限

| 权限 | 类型 | 用途 |
|---|---|---|
| `AuditLog.Read.All` | Application | 读取 Entra 审计和登录日志（必需） |
| `Directory.Read.All` | Application | 解析日志中的用户/组 GUID（可选，用于上下文富化） |

#### 3.2.4 安全厂商使用策略

Entra ID 登录日志仅保留 30 天（P1/P2），安全厂商**必须**配置持续导出以避免日志丢失。推荐通过**方案 C（诊断设置）**实时流出到 Event Hub，而非依赖 Graph API 轮询追溯。

---

### 3.3 方案 C：诊断设置（Entra ID 实时流）

#### 3.3.1 概要流程

Entra ID 支持通过"诊断设置"将日志实时推送到三类目标：

| 目标 | 适用场景 |
|---|---|
| Azure Log Analytics Workspace | 就地分析、KQL 查询、长期保留 |
| Azure Storage Account | 低成本冷存档（Immutable Blob 支持防篡改） |
| Azure Event Hub | SIEM 接入（如 Splunk、QRadar、Microsoft Sentinel） |

#### 3.3.2 可流式传输的日志类型

包括但不限于：`SignInLogs`（交互式登录）、`NonInteractiveUserSignInLogs`（非交互式）、`ServicePrincipalSignInLogs`（服务主体登录）、`AuditLogs`、`ProvisioningLogs`、`MicrosoftGraphActivityLogs`（需 P1/P2，可检测 OAuth 令牌滥用）。

#### 3.3.3 所需权限

- 配置诊断设置：需 **Security Administrator** 角色 + Azure 订阅（Event Hub / Storage）  
- 读取日志：**Reports Reader** 或 **Security Reader**

> **参考文档**：  
> - [Access activity logs in Microsoft Entra ID](https://learn.microsoft.com/en-us/entra/identity/monitoring-health/howto-access-activity-logs)  
> - [Logs available for streaming from Entra ID](https://learn.microsoft.com/en-us/entra/identity/monitoring-health/concept-diagnostic-settings-logs-options)

## 5. 影响基于日志进行告警研判的关键特性

### 5.1 日志规模与客户等级的关联

UAL 日志量**不是线性增长**的，而是受租户规模、业务模式和 License 等级驱动。

| 客户规模 | 用户数 | 预计日志量（日/条） | 关键特性与研判挑战 | 可行应对方案 |
|---|---|---|---|---|
| 小型企业（SMB） | < 300 | 数千 ~ 数十万 | 可能使用 Business License，UAL 默认关闭；事件类型受限 | 优先确认 UAL 已开启；接受全量采集 |
| 中型企业 | 300 ~ 5000 | 百万级 | E3 License 居多，无 `MailItemsAccessed`；可能存在混合 License | 分 License 等级处理事件类型差异；评估是否引导升级 E5 |
| 大型企业 / 集团 | > 5000 | 千万甚至亿级/天 | E5 Premium 事件量急剧增大；跨租户数据汇聚复杂；API 限流成为瓶颈 | 必须设计分布式采集；对 `MailItemsAccessed` 等高频事件实施预聚合后再入库 |
| 跨国集团（多租户） | 多租户汇聚 | 数倍于单租户 | 各租户独立 API 配额；需为每个租户单独注册应用和订阅 | 设计租户级并发采集架构；关注 API 配额叠加问题 |

**研判挑战举例**：`MailItemsAccessed` 对于拥有数千名 E5 用户的大型企业，每天可能产生**数亿条**记录（每封邮件被打开均产生一条）。将所有记录写入 SIEM 进行分析会产生极高成本，安全厂商通常需要在采集端做**预聚合**（如：只保留非交互式客户端、来自异地 IP 的访问记录）。

---

### 5.2 日志延迟

这是影响告警实时性最关键的特性。

| 日志类型 | 典型延迟 | 最差情况 | 对研判的影响 |
|---|---|---|---|
| UAL（Exchange、SharePoint、Teams） | **60 ~ 90 分钟** | 数小时（系统压力大时） | 无法用于实时阻断；适合事后取证和分钟级告警 |
| UAL（其他服务） | > 90 分钟 | 不保证 | 同上 |
| Entra ID 登录日志（Graph API） | 通常 < 5 分钟 | 15 分钟 | 接近实时；可用于登录异常的近实时告警 |
| Entra 诊断设置（Event Hub） | 通常 < 30 分钟 | 2 小时（多区域） | 比 Graph API 轮询更低延迟，但仍有延迟 |

**微软官方立场**（来自故障排查文档）：

> "The Office 365 Audit service doesn't guarantee a specified time when events will be delivered. Audit tries to deliver data as quickly as possible. However, Microsoft prioritizes **data completeness over latency**."

**意义**：UAL 的设计目标是**完整性优先**，而非实时性。上层研判引擎**不应将 UAL 作为需要亚分钟响应的实时告警数据源**；应将其定位为"事后确认"和"攻击链重建"的工具。实时告警应依赖 Entra 登录日志（低延迟）或 Microsoft Defender for Office 365 的实时信号。

> **参考文档**：[Troubleshooting the Management Activity API – Event availability](https://learn.microsoft.com/en-us/office/office-365-management-api/troubleshooting-the-office-365-management-activity-api)

---

### 5.3 是否支持流式拉取

**结论：UAL 不支持真正的流式拉取，本质是批量 Blob 轮询；Entra 诊断设置支持真正的流式推送。**

| 方案 | 模式 | 实现方式 | 问题 |
|---|---|---|---|
| Management Activity API（轮询） | Pull（批量） | 定期检查新 Blob，下载后处理 | 最小采集粒度为 Blob（数分钟到数十分钟），无法实现秒级感知 |
| Management Activity API（Webhook） | Push（推送） | 微软主动通知有新 Blob | 官方已记录存在通知延迟，不推荐生产使用 |
| Entra 诊断设置 → Event Hub | Push（流式） | 准实时推送到 Event Hub | 需要 Azure 订阅；Graph Activity Logs 需 P1/P2 |

**对研判的影响**：由于 UAL 采用 Blob 批量模式，告警引擎在联立跨时间窗口的事件时（如：登录 → 规则创建 → 批量下载的三步攻击链），需考虑**事件时序对齐**问题（见 §5.6）。

---

### 5.4 是否支持 API 过滤

| 方案 | 过滤能力 | 限制 |
|---|---|---|
| Management Activity API | **仅支持按 Content Type 订阅过滤**（如只订阅 Exchange 日志） | 无法在 API 层面按用户、时间、Operation 类型过滤；需业务侧全量拉取后过滤 |
| Graph API（Entra 日志） | **支持 `$filter` 服务端过滤**（按时间、用户、风险等级等） | 微软建议避免大批量调用 |
| Search-UnifiedAuditLog（PowerShell） | 支持多维度过滤（用户、时间、Operations 等） | 不适合自动化大规模采集，仅适合人工调查 |

**注意**：UAL 的 Management Activity API **不支持按 `Workload` 字段过滤**（只能在 Content Type 层面区分大类）。例如，若订阅 `Audit.General`，其中包含 Teams、Power BI 等多个服务，需在业务侧用 `Workload` 字段再次过滤。

---

### 5.5 限流策略

| License 等级 | 初始额度 | 动态调整 |
|---|---|---|
| Audit (Standard)（所有订阅默认） | 2,000 请求/分钟 | 随席位数增加动态提升 |
| Audit (Premium)（E5/A5/G5） | ~4,000 请求/分钟（约 2 倍） | 同上 |

**注意**：

- 此限制为 **2020 年前的发布者级别（Publisher-level）改为租户级别（Tenant-level）**。老的发布者级别限制意味着安全厂商若同时服务多个客户，所有客户共享同一配额。自 Audit (Premium) 引入后改为每个租户独立配额。
- 遭遇 HTTP 429 必须实现退避（Backoff），否则可能导致采集服务崩溃。
- 对大型租户的批量 Blob 下载（Content URI 请求）也受此限制约束。

> **参考文档**：[Microsoft Purview Audit (Premium) – Higher bandwidth](https://learn.microsoft.com/en-us/purview/audit-premium)

---

### 5.6 日志时序一致性（补充项 B）

这是安全运营中经常被忽视的深层问题。

**问题描述**：

M365 日志中存在多个含义不同的"时间"：

| 时间字段 | 含义 |
|---|---|
| `CreationTime`（UAL） | **事件发生时间**，是研判基准 |
| Content Blob 创建时间 | Blob 被写入 API 系统的时间，**晚于事件发生时间约 60-90 分钟** |
| `TimeGenerated`（Log Analytics） | 日志到达 Log Analytics 的时间，**非事件发生时间** |
| Entra 风险登录的 `TimeGenerated` | 风险被检测到的时间，**晚于实际登录时间** |

**对研判的具体影响**：

- 若研判引擎以**日志到达时间**（而非 `CreationTime`）作为关联基准，将导致跨日志源的事件关联失效。例如：攻击者在 08:00 登录（Entra 日志 08:02 可查），08:05 创建收件箱规则（UAL 日志 09:30 才可查），如果引擎以到达时间关联，两者将相差超过 90 分钟，无法自动关联为同一攻击链。
- **最佳实践**：所有告警引擎统一使用 `CreationTime` 作为事件时间基准，并设计支持**乱序事件的时间窗口对齐机制**（如 Flink/Spark Streaming 中的 Event Time + Watermark 模式）。

---

### 5.7 多租户日志汇聚（补充项 C）

对于安全厂商服务多个企业客户的场景，需要特别设计多租户采集架构。

| 挑战 | 描述 |
|---|---|
| 每租户独立授权 | 需为每个客户租户单独注册应用（或使用委托管理员模型）、获取 Access Token |
| API 配额独立 | 每租户 2000 请求/分钟，不可共用。大规模服务商需设计租户级并发控制 |
| 数据隔离 | 不同客户日志必须严格隔离存储，避免数据串扰 |
| 委托管理员（GDAP）模型 | 微软合作伙伴可使用 GDAP（细粒度委托管理员权限）在不成为全局管理员的情况下访问客户日志 |

**跨租户横向移动检测的挑战**：

在供应链攻击场景中（如 Solorigate），攻击者可能从被攻破的供应商 A 的租户，向客户 B 的租户发起攻击。由于两个租户的日志完全独立采集、存储，**需要在汇聚平台上进行跨租户关联分析**，这对数据架构提出了更高要求。

## 6. 安全厂商经验项总结

### 6.1 案例一：Solorigate 供应链攻击（2020）与 UAL 的盲点

**事件背景**：

2020 年 12 月，安全公司 FireEye 发现 SolarWinds Orion 平台更新包中被植入恶意代码（SUNBURST），约 18,000 家客户安装了被污染的更新。攻击者（NOBELIUM，被认为是俄罗斯 SVR）在获得端侧立足点后，向 Microsoft 365 云环境进行横向移动。

**攻击链与 UAL 的记录能力分析**：

攻击者在 M365 侧的 TTP 包括：
1. 从本地 ADFS 服务器窃取签名证书，伪造 SAML 令牌，以任意用户身份登录 M365（**UAL 中对应 `UserLoggedIn` 事件，但由于令牌合法，与正常登录无法通过日志本身区分**）
2. 在 Entra ID 中注册新的 OAuth 应用，建立持久化后门（**UAL 中对应 `Add application` 事件，可检测**）
3. 以 OAuth 应用身份访问邮箱，大量读取高管邮件（**E5 下对应 `MailItemsAccessed` 可检测；E3 下完全不可见**）
4. 修改联合域信任配置（**UAL 中对应 `Set domain authentication` 等 Entra 审计事件**）

**关键发现**：

- CISA 在其 2021 年发布的技术警报中指出：由于 UAL 默认仅保留 90 天（当时政策），**超过 90 天的攻击活动在事后溯源时已无法在 UAL 中查到**，而 Solorigate 的初始渗透最早可追溯至 2020 年 3 月，攻击潜伏期长达 9 个月。
- Entra ID 中的服务主体登录日志（Service Principal Sign-in Logs）在当时的标准配置下**需要额外开启诊断设置**才能收集，许多组织并未配置，导致关键身份凭证滥用路径无法追溯。

**对安全厂商的启示**：
- 供应链攻击的特点是"以合法凭据进行合法操作"，UAL 中的每一条记录单独来看都是正常的，攻击的异常性**只在时序关联和行为基线对比中才能体现**。
- APT 攻击的潜伏期远超大多数默认日志保留窗口，必须在**攻击发现前就已经完成日志的持续导出**，否则关键证据将永久丢失。
- 仅靠云日志无法检测 Solorigate 中的端侧行为（SUNBURST 在本地运行），**必须结合端侧 EDR 日志与云日志进行多域关联**。

> **参考文档**：
> - CISA Alert AA21-008A：[Detecting Post-Compromise Threat Activity in Microsoft Cloud Environments](https://www.cisa.gov/news-events/cybersecurity-advisories/aa21-008a)  
> - Microsoft Security Blog：[Using Microsoft 365 Defender to protect against Solorigate](https://www.microsoft.com/en-us/security/blog/2020/12/28/using-microsoft-365-defender-to-coordinate-protection-against-solorigate/)

### 6.2 案例二：OAuth 同意钓鱼与横向移动（2023-2025 持续活跃）

**事件背景**：

微软威胁情报团队在 2023 年 12 月发布报告，记录了攻击者（多组织）利用恶意 OAuth 应用进行大规模攻击的 TTP。Proofpoint 在 2025 年记录了多个国家级行为者利用 OAuth Device Code Flow 进行账户接管的活跃活动。

**攻击链**：

```
1. 攻击者发送钓鱼邮件，引导受害者授权恶意 OAuth 应用
2. 受害者点击"允许"后，攻击者获得 Access Token + Refresh Token
3. 利用 Token 访问 Graph API，读取邮件、文件
4. 在受害者租户内注册新 OAuth 应用（持久化）
5. 以高权限 OAuth 应用身份横向移动至其他用户邮箱
6. 利用获取的财务信息实施 BEC（商业邮件诈骗）
```

**UAL 中可观测到的信号**（按时序）：

| 时间点 | 可观测事件 | 所在日志类型 |
|---|---|---|
| T+0：钓鱼成功 | `Consent to application`（含高危权限） | UAL / Entra 审计 |
| T+0：Token 签发 | 非交互式登录（`NonInteractiveUserSignInLogs`） | Entra 登录日志 |
| T+30m：邮件读取 | `MailItemsAccessed`（大量 OperationCount，EWS 客户端） | UAL（仅 E5） |
| T+1h：新应用注册 | `Add application`、`Add service principal` | UAL / Entra 审计 |
| T+1h+：文件下载 | 批量 `FileDownloaded` | UAL |

**研判盲点**：

- 由于 Refresh Token 有效期通常为 90 天，攻击者可在初次 Token 获取后**长达 3 个月内**持续访问，而无需再次触发登录事件。如果研判引擎只关注登录异常，将完全错过这 3 个月的持续访问行为。
- 在 E3 License 下，没有 `MailItemsAccessed` 事件，攻击者读取了多少邮件、读取了哪些邮件，**完全不可见**。
- `Consent to application` 事件产生时，攻击者实际上使用的是受害者合法的 M365 登录来授权，因此**不会触发任何 MFA 绕过告警**（MFA 已经通过，授权本身是合法的）。

**对安全厂商的启示**：
- 针对 OAuth 攻击，必须建立**应用画像基线**：已知可信应用列表 vs. 新注册/新授权应用的权限范围对比。
- 研判引擎需要建立**Token 生命周期视角**，而非只关注登录事件：一次 Token 签发可能驱动后续数月的持续访问。
- `MicrosoftGraphActivityLogs`（Entra 诊断设置中的 Graph 活动日志）是检测 OAuth 令牌滥用的关键补充数据源，但需要 P1/P2 License 和额外配置。

> **参考文档**：
> - Microsoft Security Blog：[Threat actors misuse OAuth applications to automate financially driven attacks (2023)](https://www.microsoft.com/en-us/security/blog/2023/12/12/threat-actors-misuse-oauth-applications-to-automate-financially-driven-attacks/)  
> - Proofpoint：[Access granted: phishing with device code authorization for account takeover (2025)](https://www.proofpoint.com/us/blog/threat-insight/access-granted-phishing-device-code-authorization-account-takeover)  
> - Elastic Security Labs：[Microsoft Entra ID OAuth Phishing and Detections (2025)](https://www.elastic.co/security-labs/entra-id-oauth-phishing-detection)

### 6.3 经验项总结

**① "日志齐备"≠"安全可见"**

仅有 UAL 是不够的。攻击者在云控制面的操作是可见的，但数据面的内容（邮件正文、文件内容）、网络层行为（C2 通信、横向移动流量）以及端侧行为（进程、内存）均不可见。云日志必须与 EDR、NDR、Email Security Gateway 的日志联合使用，才能形成完整的攻击链可见性。

**② 日志保留是攻防博弈的一部分**

APT 攻击者熟知云日志的保留时长，往往将关键操作分散在默认保留期之外。安全厂商应**主动协助客户完成日志的持续导出**，并将 Entra 登录日志（仅 30 天）的导出列为最高优先级安全配置项。

**③ License 等级决定可观测深度**

E3 与 E5 客户的告警检测能力存在本质差距——前者缺少 `MailItemsAccessed` 等关键事件，这意味着安全厂商为 E3 客户提供的告警规则集必须依赖行为模式（如短时大量 `FileDownloaded`），而非精确事件。需在产品层面对不同 License 等级的检测能力有明确的能力说明。

**④ UAL 开关状态监控是必选项**

攻击者获得管理员权限后的第一步常常是关闭或削弱日志采集（如关闭 UAL、关闭特定用户的邮箱审计）。安全厂商应将 `Set-AdminAuditLogConfig`（UAL 状态变更）作为**最高优先级实时告警规则**之一，并通过定期健康检查验证日志采集是否正常。

**⑤ Token 视角比登录视角更重要**

在现代 M365 攻击中，攻击者越来越多地使用 OAuth 令牌（而非密码）进行访问，令牌有效期内的所有访问行为在日志中看起来都是"合法用户"的操作。研判引擎需要从"谁在登录"转变为"哪个应用在用什么身份访问哪些资源"，即建立**应用-用户-资源三维访问画像**。

**⑥ 时序对齐是关联分析的基础工程问题**

M365 日志在不同来源之间的延迟差异（Entra 登录 <5 分钟 vs. UAL <90 分钟）导致事件时序错位。在构建 SIEM 关联规则时，必须使用**事件发生时间**（`CreationTime`）而非日志到达时间作为关联基准，并设计足够宽的时间窗口（建议不少于 2 小时）以容纳最大延迟差异。
