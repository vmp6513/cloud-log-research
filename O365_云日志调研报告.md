## 1. 日志分类

Microsoft Entra ID（前身 Azure Active Directory）是 Microsoft 365 和 Azure 生态的统一身份平台。Entra ID 日志覆盖**身份平面**——记录"谁、何时、从哪里、以什么方式、访问了哪个资源"，以及"身份配置发生了哪些变更"。这与 Azure Activity Log 覆盖的"ARM 控制面操作"形成互补，两者合并才构成 Azure/M365 环境的完整审计视图。

Entra ID 日志与 Azure Activity Log 在以下关键维度上**完全独立**：

| 维度 | Entra ID 日志 | Azure Activity Log |
|---|---|---|
| 覆盖范围 | 身份认证 + 目录配置变更 | ARM 资源操作 |
| 默认保留 | 7 天（Free）/ 30 天（P1/P2） | 90 天 |
| 导出配置入口 | Entra Admin Center → 诊断设置 | Azure Monitor → Activity Log → 导出 |
| Log Analytics 表名 | `SigninLogs`、`AuditLogs` 等 | `AzureActivity` |

### 1.1 核心日志类别

Entra ID 通过诊断设置（Diagnostic Settings）将日志导出到外部，共支持以下类别：

| 日志类别 | Log Analytics 表名 | 默认开启 | License 要求 | 核心内容 | 安全意义 |
|---|---|---|---|---|---|
| **SignInLogs**（交互式登录） | `SigninLogs` | ✅ 收集，需配置导出 | 任意付费 License | 用户通过浏览器/应用程序直接登录的事件，包含 IP、位置、MFA 状态、条件访问结果 | **最高**，账户入侵检测的第一手数据源 |
| **NonInteractiveUserSignInLogs** | `AADNonInteractiveUserSignInLogs` | ✅ 收集，需配置导出 | 任意付费 License | 客户端应用代表用户进行的后台静默登录（如 Outlook 客户端刷新令牌），**量级通常是交互式的 5–10 倍** | 高，OAuth 令牌滥用检测的关键来源 |
| **ServicePrincipalSignInLogs** | `AADServicePrincipalSignInLogs` | ✅ 收集，需配置导出 | 任意付费 License | 应用程序/服务主体（Service Principal）的登录事件 | **高**，检测 CI/CD 凭证泄露、OAuth 应用滥用 |
| **ManagedIdentitySignInLogs** | `AADManagedIdentitySignInLogs` | ✅ 收集，需配置导出 | 任意付费 License | Azure 托管身份（Managed Identity）的认证事件 | 高，横向移动检测的关键数据（需与 Azure Activity Log 联合分析） |
| **AuditLogs** | `AuditLogs` | ✅ 收集，需配置导出 | 任意付费 License | Entra 目录的所有配置变更：用户/组/角色管理、应用注册、条件访问策略变更等 | **最高**，身份治理和权限变更审计的核心 |
| **MicrosoftGraphActivityLogs** | `MicrosoftGraphActivityLogs` | ❌ 需手动启用 | P1 / P2 | 所有对租户 Microsoft Graph API 的 HTTP 请求记录，包含调用者、操作、目标资源 | 高，检测 OAuth 令牌滥用和应用异常调用行为的补充数据源 |
| **RiskyUsers** | `AADRiskyUsers` | ❌ 需手动启用 | P2（Identity Protection） | ML 检测出的存在风险的用户账户汇总 | 高，预处理后的风险信号，可直接关联告警 |
| **UserRiskEvents** | `AADUserRiskEvents` | ❌ 需手动启用 | P2（Identity Protection） | 具体风险检测事件（如泄露凭证、匿名 IP 登录、不可能旅行等） | 高，比 RiskyUsers 更精细的风险事件溯源 |
| **CustomSecurityAttributeAuditLogs** | `AuditLogs`（子集） | ❌ 需手动启用 | 任意付费 License | 自定义安全属性的变更记录 | 中，特殊合规场景 |
| **MicrosoftServicePrincipalSignInLogs** | 独立表（预览） | ❌ 需手动启用 | 任意付费 License | 微软第一方服务之间的服务主体互相登录，**量极大，官方建议大多数客户无需开启** | 低，复杂且量大，非一般安全场景必须项 |

### 1.2 日志存储与查询方式

Entra 日志的存储与 Azure Activity Log 共享相同的目标端点，但**配置入口完全独立**：

| 存储/查询方式 | 保留时长 | 支持日志类型 | 查询能力 | 计费 |
|---|---|---|---|---|
| **Entra Admin Center 内置查询** | **30 天（P1/P2）/ 7 天（Free）** | 全部类别（门户内置视图） | 多维度过滤，不支持跨租户查询 | 免费（含在 License 中） |
| **Diagnostic Settings → Storage Account** | 由 Blob 生命周期策略决定，理论无限 | 全部类别（按诊断设置中勾选的类别） | 需配合外部工具处理 | Storage 存储费；诊断设置本身免费 |
| **Diagnostic Settings → Log Analytics Workspace** | 默认 30 天，可配置 30–730 天（Basic 表最长 12 年） | 全部类别 | **KQL 查询**，支持与 Azure Activity Log 跨表联合分析 | 按数据摄入量计费（约 $2.30/GB） |
| **Diagnostic Settings → Event Hub** | Event Hub 消息保留期：Standard 1–7 天，Premium 最长 90 天 | 全部类别 | 流式消费，实时 SIEM 接入 | Event Hub 吞吐量单元费 |
| **Microsoft Graph API** | 依赖上述存储配置（本身无存储） | SignInLogs、AuditLogs（通过 `/auditLogs/` 端点） | 支持 `$filter` 服务端过滤，但**官方不建议用于大批量采集** | Graph API 本身免费；需相应 License |

> **重要说明**：Entra 日志**默认不持久化到任何外部存储**。Entra Admin Center 内置视图中的 30 天（或 7 天）数据**到期后永久消失，且无法追溯获取**。这是 Entra 日志采集中优先级最高的配置风险项。

### 1.3 补充信号（与 Entra 日志联合使用）

| 信号类型 | 来源 | 与 Entra 日志的关系 |
|---|---|---|
| Azure Activity Log | Azure Monitor | Entra 登录日志识别了"谁登录"，Activity Log 记录了"登录后对 ARM 做了什么"——两者联合分析才能还原攻击者的完整操作链 |
| Microsoft 365 Unified Audit Log（UAL） | Microsoft Purview | Entra 登录成功后，UAL 记录邮件访问、文件下载等 M365 数据面操作（Storm-0558 案例中 MailItemsAccessed 是关键检测点） |
| Conditional Access Insights | Entra 条件访问报告 | 登录日志中 `appliedConditionalAccessPolicies` 字段记录了策略的执行结果；Insights 报告展示策略覆盖盲区 |

> - 日志集成选项：[Microsoft Entra activity log integration options](https://learn.microsoft.com/en-us/entra/identity/monitoring-health/concept-log-monitoring-integration-options-considerations)

## 2. 能力边界

### 2.1 视角边界

**身份认证行为全覆盖，认证令牌的内容与合法性不可见，数据面操作不可见。**

Entra ID 日志记录的是"认证事件"——记录谁申请了令牌、令牌是否被颁发、颁发时的条件是什么。但它不记录令牌被颁发后、持有者用令牌做了什么。

| 可见（Entra 日志中有记录） | 不可见（需其他日志补充） |
|---|---|
| 所有类型的用户登录事件（交互式、非交互式、服务主体、托管身份） | 认证令牌被颁发后的具体数据操作（需 M365 UAL 或 Azure Activity Log） |
| 登录失败的详细原因（错误码、失败原因字段） | 令牌内容本身（Payload 中的声明不被记录） |
| 每次登录使用的 MFA 方法和结果 | 攻击者在本地端侧窃取令牌的行为（需 EDR） |
| 条件访问策略对该登录的执行结果（Allow/Block/MFA Required） | 合法令牌被伪造或篡改的情况（Storm-0558 的核心盲点） |
| 所有 Entra 目录配置变更（用户创建/删除、角色分配、应用注册等） | 应用实际访问的数据内容（需各服务自己的日志） |
| 登录的来源 IP、地理位置、设备信息、客户端应用 | 网络层流量（需 Azure Network Watcher） |
| Identity Protection 的风险评估结果（P2 License） | EC2/VM 内部的进程行为（需 EDR） |
| MicrosoftGraphActivityLogs：所有对 Graph API 的 HTTP 请求 | Graph API 响应中返回的数据内容 |

**跨租户限制**：Entra 日志是**租户级别**的，一个租户的日志只覆盖该租户内发生的认证事件。攻击者使用 B2B 协作从外部租户访问时，`crossTenantAccessType` 字段会标记访问类型，但外部租户侧的操作记录在外部租户的 Entra 日志中，本租户完全看不到。

> **参考文档**：[Sign-in log activity details](https://learn.microsoft.com/en-us/entra/identity/monitoring-health/concept-sign-in-log-activity-details)

### 2.2 内容边界

**记录认证事件的元数据，不记录认证令牌的内容和数据操作的结果。**

- **登录日志**：记录了"用户成功登录了哪个应用"，但**不记录该应用被访问后返回的数据内容**。知道有人登录了 Outlook Web Access，但不知道他读了哪些邮件（需 M365 UAL 的 `MailItemsAccessed` 事件）。

- **非交互式登录的令牌刷新**：记录了"客户端应用刷新了 Access Token"，但**不记录 Refresh Token 本身的内容**。攻击者若盗取了 Refresh Token 并持续刷新令牌，每次刷新都会产生一条非交互式登录记录，但这些记录外观与合法刷新完全相同——这是 Storm-0558 案例中最核心的检测困境。

- **审计日志中的 `modifiedProperties`**：记录了属性的新旧值（`oldValue`、`newValue`），但对于**密码重置、密钥生成**等敏感操作，新值为空或被编辑（显示密码不会被记录，只记录"密码被重置"这一事实）。

- **MicrosoftGraphActivityLogs**：记录了 HTTP 请求方法、路径、响应码，但**不记录响应体（即实际返回的数据内容）**。知道有人调用了 `/v1.0/users`，但不知道返回了多少用户的详情。

- **条件访问策略评估**：登录日志中的 `appliedConditionalAccessPolicies` 记录了策略评估结果，但**不记录策略本身的详细配置内容**。

> **参考文档**：[Learn about the monitoring and health activity log schemas](https://learn.microsoft.com/en-us/entra/identity/monitoring-health/concept-activity-log-schemas)

### 2.3 日志完整性与可信度

#### 攻击者能否删除/篡改 Entra 日志？

| 操作 | 是否可行 | 所需权限 | 是否留下痕迹 | 防御手段 |
|---|---|---|---|---|
| 删除 Entra 诊断设置（停止日志导出） | **可以** | Security Administrator 角色 | ✅ 会被 Entra Audit Log 记录（`microsoft.aadiam/diagnosticsettings/delete`）；同时 Azure Activity Log 也会记录 | PIM 保护 Security Administrator 角色；Azure Policy 强制部署诊断设置 |
| 修改/篡改 Entra Admin Center 内置的 30 天日志 | **不可以** | —— | —— | Entra 平台托管，租户管理员无法修改已记录的日志 |
| 删除已导出到 Log Analytics 的日志 | **可以**（通过 Purge API） | `Data Purger` 角色（默认无人拥有） | ✅ 会被记录 | 不授予 Data Purger 角色；对 Workspace 设置 Resource Lock |
| 删除已导出到 Storage Account 的日志 Blob | **可以**（若无 Object Lock） | Storage Blob 写权限 | ❌ 不主动告警 | Immutable Blob（WORM）策略；独立的日志归档订阅 |
| 创建虚假的 Entra 日志条目 | **不可以** | —— | —— | Entra 日志由平台生成，管理员无法注入虚假记录 |

**关键结论**：Entra 内置的 30 天日志**不可篡改也不可删除**（平台托管）。真正的风险在于攻击者**停止日志导出**（删除诊断设置）和**删除已导出到外部存储的历史日志**。

**最佳实践**：
1. 通过 **Azure Policy** 自动强制在所有订阅中部署 Entra 诊断设置（Deploy If Not Exists 策略），防止手动删除后失去持久化
2. 将 Entra 日志导出至**独立的日志归档订阅**中的 Storage Account，并开启 **Immutable Blob（WORM）策略**
3. 对诊断设置删除操作（`microsoft.aadiam/diagnosticsettings/delete`）设置实时告警

> **参考文档**：
> - [Integrate Microsoft Entra logs with Azure Monitor logs](https://learn.microsoft.com/en-us/entra/identity/monitoring-health/howto-integrate-activity-logs-with-azure-monitor-logs)
> - [Archive Microsoft Entra logs to an Azure storage account](https://learn.microsoft.com/en-us/entra/identity/monitoring-health/howto-archive-logs-to-storage-account)

### 2.4 日志保留策略与取证窗口

| 存储方式 | 默认保留 | 最长保留 | 注意事项 |
|---|---|---|---|
| Entra Admin Center 内置 | **30 天（P1/P2）/ 7 天（Free）** | 30 天（固定不可延长） | **过期后永久不可追溯**——这是 Entra 日志中最高优先级的风险点 |
| Diagnostic Settings → Storage Account | 由 Blob 生命周期策略决定 | 理论无限 | **需提前配置**，过期前未导出的日志无法补救 |
| Diagnostic Settings → Log Analytics Workspace | 默认 30 天；可配 30–730 天；Basic 表最长 12 年 | 12 年 | 按摄入量计费；非交互式 + 服务主体日志量极大，成本需评估 |
| Diagnostic Settings → Event Hub | Standard：1–7 天；Premium：最长 90 天 | 90 天 | 仅流式传输，不适合长期存档 |

**高危场景：APT 潜伏期 vs. 30 天日志窗口**

Entra 登录日志仅默认保留 30 天（P1/P2）。APT 攻击者（如 Storm-0558）在环境中的潜伏期可长达数月甚至数年。若安全团队在攻击发生 31 天后才发现入侵，**所有攻击者的登录记录已永久消失**，无法再追溯攻击者使用哪些账号、从哪些 IP、在哪个时间点登录过系统。

Storm-0558 的调查报告明确指出："due to log retention policies, we don't have logs with specific evidence of this exfiltration by this actor"——连微软自身都因日志保留策略不足，无法完整还原攻击路径。

**安全厂商的行动建议**：将"验证 Entra 诊断设置已配置并指向持久存储（Storage Account 或 Log Analytics）"列为客户接入时的**第一步强制检查项**，优先级高于任何其他安全配置。

### 2.5 其他关键限制

**① License 等级决定可见的日志类别和风险能力，而非仅仅影响保留期**

这是最常见的误区。不同 License 等级对 Entra 日志的影响如下：

| 功能 | Free | P1 | P2 |
|---|---|---|---|
| 登录日志门户查看 | ✅（7 天） | ✅（30 天） | ✅（30 天） |
| 诊断设置导出所有登录类别 | ❌ | ✅ | ✅ |
| MicrosoftGraphActivityLogs | ❌ | ✅ | ✅ |
| RiskyUsers / UserRiskEvents（Identity Protection） | ❌ | ❌（只有基础检测） | ✅（完整风险信号） |
| 条件访问策略（影响日志中 CA 字段的完整性） | ❌ | ✅ | ✅ |

**② 非交互式登录日志量是交互式的 5–10 倍，是成本规划的核心变量**

微软官方文档明确指出："non-interactive user sign-ins and service principal sign-ins can be 5 to 10 times larger than the interactive user sign-ins"。对于 Microsoft 365 重度使用的企业，Teams、Exchange、SharePoint 的每次后台令牌刷新都产生一条非交互式登录记录，这些记录的总量远超用户主动登录的量级。在规划 Event Hub TU 数量和 Log Analytics 摄入成本时，这是最容易被低估的量级放大因素。

**③ Graph API 不适合大批量采集——官方推荐诊断设置路径**

Microsoft Graph API（`/auditLogs/signIns`）支持 `$filter` 服务端过滤，但微软明确在文档中指出大批量数据应通过诊断设置（Event Hub/Storage）路由，而非 Graph API 轮询。Graph API 的限制：每次请求最多返回 1000 条；分页通过 `@odata.nextLink` 实现；高并发时可能遭遇限流（429）；无法实时感知新事件（需轮询）。建议 Graph API 仅用于**人工调查**或**补偿扫描**（对历史日志的按需查询），不作为后台批量采集的主路径。

**④ 部分非交互式登录在早期会出现在交互式日志中（历史遗留问题）**

微软 FAQ 文档确认，部分使用 FIDO2 密钥的非交互式登录仍会出现在 `SigninLogs`（交互式）表中，而非 `AADNonInteractiveUserSignInLogs`。这是一个历史遗留行为，会导致后台基于表名分类处理时产生计数偏差，解析时需以记录内的 `isInteractive` 字段为准，而非以 Event Hub 中的消息类别（`category`）字段为准。

**⑤ 诊断设置将日志批量打包，延迟约 5 分钟**

微软文档说明：Entra 日志通过诊断设置导出时，**事件被批量打包，约每 5 分钟推送一批**（"Events are batched into about 5-minute intervals"）。这意味着即使走 Event Hub 路径，Entra 日志的最小粒度延迟也是约 5 分钟，不能实现秒级实时。

**⑥ AuditLogs 中的 IP 地址位于嵌套字段，解析需特别注意**

审计日志（`AuditLogs`）中，操作发起者的 IP 地址位于 `initiatedBy.user.ipAddress` 的嵌套路径下，而不是顶层字段。在 Log Analytics KQL 中需用 `InitiatedBy.user.ipAddress` 提取，在 Event Hub 消息体中需按 JSON 嵌套路径解析。登录日志（`SigninLogs`）则在顶层提供 `IPAddress` 字段。这种字段位置的不一致是多表关联分析时的常见解析错误。

## 3. 日志采集流程

### 3.0 章节背景

**采集后台定位**：我们的后台程序部署在独立 SaaS 平台，不进入客户的 Azure/Entra 环境。当客户接入租户后，后台通过以下方式主动拉取 Entra 日志：

- **Push-Pull（流式消费）**：客户配置诊断设置将 Entra 日志推送到 Event Hub，后台作为 Event Hub Consumer 消费消息
- **Pull（存储拉取）**：客户配置诊断设置将 Entra 日志归档到 Storage Account，后台定时扫描增量文件

**准实时目标**：5–15 分钟端到端延迟（受 5 分钟批量打包延迟约束，理论下限约为 6–10 分钟）。

---

### 3.1 授权模型

Entra 日志的授权模型与 Azure Activity Log 报告中的描述相同，采用**服务主体（Service Principal）**方式，但读取 Entra 日志所需权限与 Activity Log 不同。

| 模型 | 方式描述 | 推荐程度 | 适用场景 |
|---|---|---|---|
| **模型 A：多租户应用 + Federated Credential** | 客户对 SaaS 平台的多租户应用执行 Admin Consent；后台使用 Workload Identity Federation 获取临时令牌 | ✅ **强烈推荐** | 大多数场景；无长期密钥风险 |
| **模型 B：单租户应用 + Client Secret** | 客户注册专用应用，提供 Client Secret；有效期 ≤ 1 年，需定期轮换 | ⚠️ 次选 | 合规限制无法使用多租户应用 |

**Entra 日志读取所需的最小 API 权限**（配置在应用注册的 API 权限中）：

| 权限 | 类型 | 用途 |
|---|---|---|
| `AuditLog.Read.All` | Application | 读取审计日志和所有类型的登录日志（必需） |
| `Directory.Read.All` | Application | 解析日志中的用户/组对象 ID 为可读名称（可选，用于日志富化） |

> **注意**：读取 Event Hub 或 Storage Account 的权限（`Azure Event Hubs Data Receiver` / `Storage Blob Data Reader`）与上述 Graph API 权限独立，均需单独授权。若后台通过 Event Hub/Storage 消费日志，则不需要 `AuditLog.Read.All`；若后台直接调用 Graph API 补偿查询，则需要此权限。

---

### 3.2 采集方案总览与选型建议

| 方案 | 核心链路 | 端到端延迟 | 客户配置复杂度 | 客户侧增量成本 | 推荐场景 |
|---|---|---|---|---|---|
| **方案一**：诊断设置 → Event Hub（主流方案） | Entra 日志 → Diagnostic Settings → Event Hub → 后台消费 | **6–15 分钟** | ⭐⭐（中，约 3 步） | Event Hub 吞吐量单元费（注意非交互式日志量大） | 绝大多数需要准实时告警的客户 |
| **方案二**：诊断设置 → Storage Account（存档方案） | Entra 日志 → Diagnostic Settings → Storage Account → 后台轮询 | **10–30 分钟** | ⭐（低，约 2 步） | Storage 存储费（低） | 对实时性要求低；最简单配置；合规存档 |
| **方案三**：方案一 + 方案二（双轨方案） | Event Hub 实时流 + Storage Account 兜底存档 | 实时路径 6–15 分钟 | ⭐⭐⭐（较高） | 两套成本之和 | 日志完整性要求高的客户（APT 溯源场景） |

> 方案一和方案二均兼容授权模型 A 和模型 B。

**业界使用情况参考**：

| 方案 | 厂商使用案例 | 文档链接 |
|---|---|---|
| **方案一**（诊断设置→Event Hub） | **Splunk**：官方 Splunk Add-on for Microsoft Cloud Services 通过 Event Hub 接入 Entra 登录日志和审计日志，这是 Microsoft 官方推荐的 SIEM 集成路径 | [Stream Microsoft Entra logs to an event hub](https://learn.microsoft.com/en-us/entra/identity/monitoring-health/howto-stream-logs-to-event-hub) |
| **方案一**（诊断设置→Event Hub） | **IBM QRadar / ArcSight / SumoLogic**：均在微软官方文档中被列为 Event Hub 的支持合作伙伴 SIEM | [Microsoft Entra activity log integration options](https://learn.microsoft.com/en-us/entra/identity/monitoring-health/concept-log-monitoring-integration-options-considerations) |
| **方案一**（诊断设置→Event Hub） | **Microsoft Sentinel**：原生支持通过 Event Hub 或 Workspace 消费 Entra 日志，并内置多条 Entra 相关告警规则 | [Connect Microsoft Entra ID to Microsoft Sentinel](https://learn.microsoft.com/en-us/azure/sentinel/connect-azure-active-directory) |

**选型决策树**：

```
是否需要 ≤ 15 分钟准实时告警？
    ├─ 是 → 方案一（Event Hub）；对日志完整性要求极高则选方案三
    └─ 否 → 方案二（Storage 轮询，最简单配置，适合合规存档）
```

### 3.3 方案一：诊断设置 → Event Hub（主流方案）

#### 3.3.1 整体链路

```
[Entra ID 认证事件发生]
      ↓ Entra 内部批量打包（约 5 分钟）
[Diagnostic Settings：将日志批次推送到 Event Hub]
      ↓ 秒级入队
[Event Hub Namespace / Event Hub]
      ↓ 后台长连接消费（AMQP）
[SaaS 采集后台：解析 JSON → 入库]
```

端到端延迟：Entra 批量打包（~5 分钟）+ Event Hub 消费（秒级）= **约 6–15 分钟**。

> **与 Azure Activity Log 的区别**：Azure Activity Log 的诊断延迟通常是秒级，而 Entra 日志存在约 5 分钟的批量打包窗口，这是 Entra 日志端到端延迟的下限，无法通过任何配置压缩。

#### 3.3.2 客户需要配置的内容

> **重要前提**：Entra 诊断设置需要 **Security Administrator** 角色。这与配置 Azure 订阅诊断设置所需的 Owner/Contributor 角色不同，需要提前与客户确认操作权限。

---

**前置步骤：确认 Event Hub 已就绪**

若已为 Azure Activity Log 创建了 Event Hub Namespace，可复用该 Namespace，但应为 Entra 日志单独创建一个 Event Hub（如命名为 `entra-logs`），以与 Activity Log 隔离，避免后台解析混淆。

若尚无 Event Hub，参照 Azure Activity Log 调研报告中的 Event Hub 创建步骤，或使用以下 ARM Template 快速创建：

```json
{
  "type": "Microsoft.EventHub/namespaces/eventhubs",
  "name": "[concat(parameters('namespaceName'), '/entra-logs')]",
  "apiVersion": "2022-01-01-preview",
  "properties": {
    "partitionCount": 4,
    "messageRetentionInDays": 7
  }
}
```

---

**第 1 步：配置 Entra ID 诊断设置**

控制台操作（约 5 分钟）：

1. 打开 **Entra Admin Center**（entra.microsoft.com）→ 左侧菜单「Identity」→「Monitoring & health」→「Diagnostic settings」
2. 点击「+ Add diagnostic setting」
3. 诊断设置名称：填写（如 `entra-logs-to-eventhub`）
4. 日志类别：勾选以下类别（**最小必选集**）：
   - ✅ `AuditLogs`（目录操作审计，必选）
   - ✅ `SignInLogs`（交互式登录，必选）
   - ✅ `NonInteractiveUserSignInLogs`（非交互式登录，**量大，必须纳入规划**）
   - ✅ `ServicePrincipalSignInLogs`（服务主体登录，必选）
   - ✅ `ManagedIdentitySignInLogs`（托管身份登录，必选）
   - ⬜ `MicrosoftGraphActivityLogs`（按需开启，需 P1/P2，量大）
   - ⬜ `RiskyUsers` / `UserRiskEvents`（按需开启，需 P2）
5. 目标详细信息：勾选「Stream to an event hub」，选择对应的 Namespace 和 `entra-logs` Event Hub
6. 点击「Save」

> ⚠️ **关键注意事项**：
> - Entra 诊断设置是**租户级别**的（不是订阅级别）。一个租户只需配置一次，即覆盖该租户下所有订阅中的用户和服务主体的身份行为。
> - 修改此设置需要 **Security Administrator** 角色，普通 Contributor 无权操作。

ARM Template（通过 Azure Resource Manager 部署）：

```json
{
  "type": "microsoft.aadiam/diagnosticSettings",
  "apiVersion": "2017-04-01",
  "name": "entra-logs-to-eventhub",
  "properties": {
    "eventHubAuthorizationRuleId": "[parameters('eventHubAuthRuleId')]",
    "eventHubName": "entra-logs",
    "logs": [
      { "category": "AuditLogs",                       "enabled": true },
      { "category": "SignInLogs",                       "enabled": true },
      { "category": "NonInteractiveUserSignInLogs",     "enabled": true },
      { "category": "ServicePrincipalSignInLogs",       "enabled": true },
      { "category": "ManagedIdentitySignInLogs",        "enabled": true }
    ]
  }
}
```

**第 2 步：授权 SaaS 后台读取 Event Hub**

```bash

az role assignment create \
  --assignee {saas-service-principal-object-id} \
  --role "Azure Event Hubs Data Receiver" \
  --scope /subscriptions/{sub-id}/resourceGroups/{rg}/providers/Microsoft.EventHub/namespaces/{namespace}/eventhubs/entra-logs
```

**第 3 步：在 SaaS 平台填写接入信息**

| 字段 | 内容 |
|---|---|
| 租户 ID（Tenant ID） | 客户的 Entra 租户 ID（GUID） |
| Event Hub Namespace | 客户创建的 Namespace 名称 |
| Event Hub 名称（Entra 日志） | `entra-logs` |
| 授权方式 | 模型 A：服务主体 Object ID；模型 B：App ID + Client Secret |

**Storage Account 的文件路径格式**：


#### 3.3.4 关键字段

**登录日志（SignInLogs / AADNonInteractiveUserSignInLogs / ServicePrincipalSignInLogs）**：

| 字段（在 `properties` 内） | 类型 | 说明 | 研判用途 |
|---|---|---|---|
| `createdDateTime` | string (ISO 8601 UTC) | **事件发生时间**，研判基准；消息外层的 `time` 是批次推送时间，不是事件时间 | 必须以此为时间基准 |
| `id` | GUID | 登录事件唯一标识 | 去重基准 |
| `userPrincipalName` | string | 登录的用户 UPN（Email 格式） | 跨日志类型关联的核心身份标识 |
| `userId` | GUID | 用户的 Entra Object ID | 不会因用户名变更而改变，更稳定的关联键 |
| `appId` | GUID | 被登录的应用程序 App ID | 非白名单 AppId 是 OAuth 应用滥用的关键信号 |
| `appDisplayName` | string | 应用程序显示名称 | 辅助识别，但可被伪造；应优先使用 `appId` |
| `ipAddress` | string | 登录来源 IP | 地理位置异常检测的核心字段 |
| `clientAppUsed` | string | 使用的客户端类型（Browser / IMAP / Exchange ActiveSync 等） | 遗留协议（如 IMAP、POP3）绕过 MFA，是重要攻击向量 |
| `conditionalAccessStatus` | string | 条件访问策略执行结果（success / failure / notApplied） | `notApplied` 说明未被任何 CA 策略覆盖，是配置盲区 |
| `authenticationRequirement` | string | 所需的认证强度（singleFactorAuthentication / multiFactorAuthentication） | `singleFactorAuthentication` + 敏感应用 = 高风险 |
| `riskLevelAggregated` | string | Identity Protection 给出的综合风险等级（none / low / medium / high） | 需 P2 License；`high` 需立即响应 |
| `riskDetail` | string | 风险的具体原因（如 `userPassedMFADrivenByRiskBasedPolicy`） | 研判风险处置逻辑的依据 |
| `isInteractive` | boolean | 是否是交互式登录 | 区分交互式和非交互式的真正依据（不能依赖日志类别） |
| `status.errorCode` | integer | 登录失败的错误码（0 = 成功；50126 = 密码错误；50074 = MFA 要求等） | 密集的特定错误码组合是攻击模式的指示器 |
| `crossTenantAccessType` | string | 跨租户访问类型（none / b2bCollaboration / b2bDirectConnect / microsoftSupport） | 外部身份访问内部资源的检测 |
| `location.countryOrRegion` | string | 登录的地理位置（国家） | 异地登录检测；注意 VPN 和 Tor 的地理位置混淆 |

**审计日志（AuditLogs）**：

| 字段 | 类型 | 说明 | 研判用途 |
|---|---|---|---|
| `properties.activityDateTime` | string | 操作发生时间；消息外层 `time` 是批次推送时间 | 时间基准 |
| `properties.id` | string | 审计事件唯一标识 | 去重基准 |
| `properties.activityDisplayName` | string | 操作的可读名称（如 `Add member to role`、`Reset user password`） | 告警规则的核心过滤字段 |
| `properties.category` | string | 操作所属类别（UserManagement / GroupManagement / RoleManagement / ApplicationManagement / Policy 等） | 快速分类过滤 |
| `properties.initiatedBy.user.userPrincipalName` | string | 发起操作的用户 UPN | 身份溯源核心字段；**注意嵌套路径** |
| `properties.initiatedBy.user.ipAddress` | string | 发起者的 IP | **注意嵌套路径**，与登录日志顶层 `ipAddress` 字段位置不同 |
| `properties.initiatedBy.app.appId` | string | 若由应用发起操作，记录应用的 App ID | 自动化工具或攻击者使用的应用的标识 |
| `properties.targetResources[].type` | string | 被操作的目标资源类型（User / Group / Role / Application / Policy 等） | 过滤高价值目标（如 Role）的操作 |
| `properties.targetResources[].displayName` | string | 被操作的目标资源名称 | 如被分配的角色名称 |
| `properties.targetResources[].modifiedProperties[]` | array | 被修改的属性列表（含 `oldValue` 和 `newValue`） | 权限变更溯源的关键数据 |
| `properties.result` | string | 操作结果（success / failure） | 失败的批量操作是权限探测信号 |
| `properties.loggedByService` | string | 记录此事件的 Entra 服务（Core Directory / PIM / Conditional Access 等） | 区分操作来源 |

#### 3.3.6 踩坑与注意事项

**① 诊断设置是租户级别，不是订阅级别，只需配置一次**

Entra 诊断设置覆盖整个租户，一次配置即可捕获该租户下**所有订阅、所有用户、所有应用**的身份事件。这与 Azure Activity Log 诊断设置（每个订阅单独配置）不同。客户可能误以为需要为每个订阅分别配置，导致重复配置，但实际上 Entra 端只需一套。

**② 时间字段有两层：外层 `time` 是批次推送时间，内层 `properties.createdDateTime` 是事件时间**

Event Hub 消息外层的 `time` 字段是该批次被推送到 Event Hub 的时间，与内层 `properties.createdDateTime`（实际发生时间）相差约 5 分钟。研判引擎**必须使用 `properties.createdDateTime`**（SignInLogs）或 `properties.activityDateTime`（AuditLogs）作为事件时间基准，而非外层 `time`。

**③ 非交互式登录量极大，Event Hub TU 规划不能参照交互式登录估算**

启用 `NonInteractiveUserSignInLogs` 后，Event Hub 吞吐量需求会大幅上升。建议：初始配置时开启 Auto-inflate（自动扩容）；在正式接入前采集 1–2 天样本评估流量；监控 Event Hub 的 `IncomingMessages` 和 `ThrottledRequests` 指标，一旦出现限流立即扩容。

**④ 不同类别的日志在 Event Hub 中混在同一消息流，需按 `category` 字段分流解析**

所有 Entra 日志类别（SignInLogs、AuditLogs、ServicePrincipalSignInLogs 等）均写入同一个指定的 Event Hub，**不会按类别自动分发到不同的 Event Hub**。后台消费时必须先检查每条记录的 `category` 字段，再按对应 Schema 解析，否则混用字段会导致解析错误。

**⑤ 诊断设置被删除是高危告警点**

若攻击者删除了 Entra 诊断设置（`microsoft.aadiam/diagnosticsettings/delete`），后续的所有身份认证事件将停止导出到 SIEM，攻击者之后的登录和操作完全不可见。这条操作本身会被 Entra Audit Log 记录（在删除发生时），**是"最后被记录的高危操作"**，需设置最高优先级实时告警。

> - [Integrate Microsoft Entra logs with Azure Monitor logs](https://learn.microsoft.com/en-us/entra/identity/monitoring-health/howto-integrate-activity-logs-with-azure-monitor-logs)

### 3.4 方案二：诊断设置 → Storage Account（存档方案）

**与方案一的核心区别**：

| 对比项 | 方案一（Event Hub） | 方案二（Storage Account） |
|---|---|---|
| 延迟 | 6–15 分钟 | 10–30 分钟（文件按小时切分） |
| 客户配置步骤 | 约 3 步 | **约 2 步（仅需 Storage）** |
| 积压丢失 | 消息超期（1–7 天）永久丢失 | **Storage 文件持久保存，无丢失风险** |
| 历史日志补救 | 无（Event Hub 无兜底） | **有**（文件持久存在，可补偿扫描） |
| 成本 | Event Hub TU 费 | 仅 Storage 存储费（更低） |

#### 3.4.1 客户配置步骤

**第 1 步**：创建或复用 Storage Account，建议开启 Immutable Blob（WORM）策略防止历史日志被删除。

**第 2 步**：在 Entra Admin Center 配置诊断设置，目标选「Archive to storage account」，其余类别勾选与方案一相同。

**第 3 步**：授权 SaaS 后台读取 Storage：

```bash
az role assignment create \
  --assignee {saas-service-principal-object-id} \
  --role "Storage Blob Data Reader" \
  --scope /subscriptions/{sub-id}/resourceGroups/{rg}/providers/Microsoft.Storage/storageAccounts/{account}
```

后台按上文 §3.3.3 中描述的路径格式进行小时级增量扫描。

---

### 3.5 方案三：方案一 + 方案二（双轨方案）

同时配置 Event Hub（实时消费）和 Storage Account（持久存档）。在 Azure 诊断设置中，单个诊断设置最多可同时指定 5 个目标，因此可在同一个诊断设置中同时指定 Event Hub 和 Storage Account，客户只需操作一次。

**核心价值**：Event Hub 消息超期（1–7 天）丢失后，Storage Account 中的文件永久保留，后台可通过补偿扫描追回漏拉的历史数据。这对需要完整溯源能力（如 APT 调查）的场景尤为重要。

---

### 3.6 方案对比与选型建议

| 维度 | 方案一（Event Hub） | 方案二（Storage 轮询） | 方案三（双轨） |
|---|---|---|---|
| **延迟** | 6–15 分钟 | 10–30 分钟 | 实时路径 6–15 分钟 |
| **客户配置步骤** | 3 步 | 2 步，最简单 | 3 步（两个目标合并在一个诊断设置中） |
| **日志类型覆盖** | Entra 全类别 | Entra 全类别 | Entra 全类别 |
| **历史日志留存** | ❌ 无（消息超期即删） | ✅ 有（Blob 持久存储） | ✅ 有 |
| **积压导致丢失** | **不可补救**（超期 = 永久丢失；无 DLQ 机制） | **无此风险**（Blob 持久，仅影响延迟） | **可补救**（Storage 兜底） |
| **采集侧过滤** | ❌ 全量推送；可在诊断设置中按类别过滤 | ❌ 同左 | ❌ 同左 |
| **客户侧增量成本** | Event Hub TU 费（注意非交互式日志量） | Storage 存储费（极低） | 两者之和（仍可接受） |

**后台工程建议（针对方案一）**：
- **预防积压**：监控 Event Hub `IncomingMessages` 和 Consumer Lag；非交互式日志高峰时段（工作日上班时间）流量显著高于夜间，TU 扩缩容应有足够余量
- **失败处理**：持久化 Checkpoint（消费位点），后台重启后从 Checkpoint 继续消费，不丢失已推送未消费的消息
- **最终一致（仅方案三）**：后台定期对 Storage 做增量路径扫描，与已处理文件列表对比，补拉漏拉的日志文件

> **参考文档**：
> - [Stream Microsoft Entra logs to an event hub](https://learn.microsoft.com/en-us/entra/identity/monitoring-health/howto-stream-logs-to-event-hub)
> - [Archive Microsoft Entra logs to an Azure storage account](https://learn.microsoft.com/en-us/entra/identity/monitoring-health/howto-archive-logs-to-storage-account)
> - [Azure Event Hubs Auto-inflate](https://learn.microsoft.com/en-us/azure/event-hubs/event-hubs-auto-inflate)

**研判意义**：三个高危信号同时出现：`authenticationRequirement: singleFactorAuthentication`（未使用 MFA）、`conditionalAccessStatus: notApplied`（未被任何条件访问策略保护，说明存在 CA 覆盖盲区）、`location.countryOrRegion: RU`（异地登录）。`appliedConditionalAccessPolicies: []`（空数组）是一个关键的安全配置盲区信号，说明该用户的该次登录完全未受到条件访问策略的保护，无 MFA 要求、无设备合规要求、无 IP 范围限制。

### 4.2 登录失败——密码错误高频（密码喷射攻击信号）

**触发方式**：对同一用户账号在短时间内使用多个不同密码尝试登录

```json
{
  "time": "2024-03-15T09:05:12.123Z",
  "category": "SignInLogs",
  "properties": {
    "id": "signin-fail-001",
    "createdDateTime": "2024-03-15T09:05:12.123Z",
    "userPrincipalName": "bob@contoso.com",
    "userId": "user-object-id-bob",
    "appId": "4765445b-32c6-49b0-83e6-1d93765276ca",
    "appDisplayName": "Microsoft Office",
    "ipAddress": "198.51.100.23",
    "clientAppUsed": "Exchange ActiveSync",
    "conditionalAccessStatus": "notApplied",
    "authenticationRequirement": "singleFactorAuthentication",
    "isInteractive": false,
    "riskLevelAggregated": "none",
    "status": {
      "errorCode": 50126,
      "failureReason": "Invalid username or password or Invalid on-premise username or password.",
      "additionalDetails": "Check your username and password and try again."
    },
    "location": { "countryOrRegion": "US" }
  }
}
```

**研判意义**：`errorCode: 50126` 表示凭证错误，是密码喷射（Password Spray）和暴力破解的核心信号。需注意：`clientAppUsed: Exchange ActiveSync` 是遗留认证协议，**无法被 MFA 保护**，是攻击者绕过 MFA 策略的经典路径。研判规则应统计：同一 IP 在 5 分钟内对 20+ 个不同用户账号产生 50126 错误，这是密码喷射的统计特征（不针对单个用户高频，而是对多用户分散尝试）。

### 4.3 服务主体登录——异常 IP 和非预期资源访问（ServicePrincipalSignInLogs）

**触发方式**：攻击者使用泄露的 Client Secret 以服务主体身份登录，访问 Microsoft Graph（需开启 ServicePrincipalSignInLogs）

```json
{
  "time": "2024-03-15T11:22:00.000Z",
  "category": "ServicePrincipalSignInLogs",
  "properties": {
    "id": "sp-signin-001",
    "createdDateTime": "2024-03-15T11:22:00.000Z",
    "servicePrincipalId": "sp-object-id",
    "servicePrincipalName": "MyApp@contoso.com",
    "appId": "malicious-or-compromised-app-id",
    "resourceId": "00000003-0000-0000-c000-000000000000",
    "resourceDisplayName": "Microsoft Graph",
    "ipAddress": "198.51.100.77",
    "conditionalAccessStatus": "notApplied",
    "status": { "errorCode": 0 },
    "location": { "countryOrRegion": "CN" },
    "riskLevelAggregated": "none"
  }
}
```

**研判意义**：服务主体登录成功（`errorCode: 0`），访问了 Microsoft Graph（`resourceDisplayName: Microsoft Graph`，包含整个 M365 数据访问能力），来源 IP 位于异常地理位置。这是 OAuth 应用凭证泄露（CI/CD 的 Client Secret 被盗用）或应用被注册为恶意后门（OAuth 钓鱼攻击）的典型信号。需对比历史基线：该 Service Principal 是否通常从这个 IP 范围登录？是否通常访问这个资源？

### 4.4 审计日志——高特权角色分配（AuditLogs）

**触发方式**：攻击者获得全局管理员权限后，将自己的账号添加到 Global Administrator 角色

```json
{
```

### 4.5 审计日志——OAuth 应用同意（OAuth 钓鱼攻击）

**触发方式**：用户受钓鱼邮件诱导，为恶意第三方应用授予了高权限 OAuth 同意（Consent to application）

```json
{
  "time": "2024-03-15T16:05:44.000Z",
  "category": "AuditLogs",
  "properties": {
    "id": "audit-consent-001",
    "activityDateTime": "2024-03-15T16:05:44.000Z",
    "activityDisplayName": "Consent to application",
    "category": "ApplicationManagement",
    "result": "success",
    "loggedByService": "Core Directory",
    "initiatedBy": {
      "user": {
        "userPrincipalName": "victim@contoso.com",
        "ipAddress": "203.0.113.99"
      }
    },
    "targetResources": [
      {
        "id": "malicious-app-sp-object-id",
        "displayName": "Totally Legit App",
        "type": "ServicePrincipal",
        "modifiedProperties": [
          {
            "displayName": "ConsentContext.IsAdminConsent",
            "oldValue": null,
            "newValue": "\"false\""
          }
        ]
      }
    ],
    "additionalDetails": [
      { "key": "AppId", "value": "malicious-app-id-xxxx" },
      { "key": "Permissions", "value": "Mail.Read Files.ReadWrite.All User.Read" }
    ]
  }
}
```

**研判意义**：`activityDisplayName: Consent to application` + `additionalDetails.Permissions` 中包含 `Mail.Read`、`Files.ReadWrite.All` 等高权限，且 `ConsentContext.IsAdminConsent: false`（是用户个人同意，而非管理员同意），是 OAuth 钓鱼攻击成功的核心信号。攻击者在此事件发生后即可使用该应用的权限持续访问用户邮件和文件，无需再触发任何新的登录事件（令牌在有效期内自动刷新），导致后续活动在登录日志中难以识别。

### 4.6 非交互式登录——异常 Refresh Token 刷新（后台静默访问）

**触发方式**：攻击者持有盗取的 Refresh Token，后台静默刷新 Access Token（需开启 NonInteractiveUserSignInLogs）

```json
{
  "time": "2024-03-15T20:00:00.000Z",
  "category": "NonInteractiveUserSignInLogs",
  "properties": {
    "id": "noninteractive-001",
    "createdDateTime": "2024-03-15T20:00:00.000Z",
    "userPrincipalName": "victim@contoso.com",
    "userId": "victim-user-id",
    "appId": "malicious-app-id-xxxx",
    "appDisplayName": "Totally Legit App",
    "resourceId": "00000002-0000-0000-c000-000000000000",
    "resourceDisplayName": "Office 365 Exchange Online",
    "ipAddress": "198.51.100.55",
    "clientAppUsed": "Browser",
    "isInteractive": false,
    "conditionalAccessStatus": "notApplied",
    "authenticationRequirement": "singleFactorAuthentication",
    "status": { "errorCode": 0 },
    "location": { "countryOrRegion": "RU" }
  }
}
```

**研判意义**：这是 OAuth 钓鱼攻击（§4.5）成功后的持续访问阶段。攻击者使用被授权的应用的 Refresh Token，每隔约 1 小时静默刷新一次 Access Token，产生非交互式登录记录。外观上与正常的后台令牌刷新完全相同，仅通过以下异常特征可识别：`appId` 与 §4.5 中的恶意应用一致、`ipAddress` 持续来自非预期地理位置（`countryOrRegion: RU`）、`resourceDisplayName: Office 365 Exchange Online`（说明正在持续访问邮件）。Refresh Token 默认有效期 90 天，意味着攻击者可在不触发任何新的身份验证的情况下，持续访问长达 3 个月。

## 5. 影响基于日志进行告警研判的关键特性

### 5.1 日志规模与客户等级的关联

Entra 日志的量级主要由**租户用户数**、**M365 服务使用强度**和**开启的日志类别**决定，与 Azure 订阅数量关系不大（Entra 日志是租户级别的）。

#### 单租户日志量参考

| 租户特征 | 用户数 | 交互式登录量（日/条） | 非交互式登录量（日/条） | 服务主体登录量（日/条） | 审计日志量（日/条） |
|---|---|---|---|---|---|
| 轻量使用（以邮件为主） | < 300 | 数千 ~ 1 万 | **3 万 ~ 10 万**（5–10 倍） | 数百 ~ 数千 | 数百 |
| 标准 M365 使用（Teams/SharePoint 密集） | 300 ~ 5000 | 数万 | **数十万 ~ 百万**（Teams 令牌刷新频繁） | 数千 ~ 数万 | 数千 ~ 数万 |
| 大型企业（高自动化 + 多应用） | > 5000 | 数十万 | **百万 ~ 千万**（CI/CD + 服务账号密集） | 数万 ~ 百万 | 数万 ~ 数十万 |

> **关键数据**：微软官方文档明确指出，非交互式登录和服务主体登录的数据量通常是交互式用户登录的 **5 到 10 倍**（[来源](https://learn.microsoft.com/en-us/entra/identity/monitoring-health/howto-stream-logs-to-event-hub)）。微软建议在正式接入前采集 1–2 天样本，用 Azure 定价计算器估算实际成本。

#### 按客户规模的整体分层

| 客户规模 | 租户数 | 用户规模 | 后台日汇聚量（全类别 Entra 日志） | 关键研判挑战 |
|---|---|---|---|---|
| 小型企业 | 1 | < 300 人 | 数万 ~ 十万 | License 可能不足（Free 仅 7 天）；诊断设置未配置 |
| 中型企业 | 1–2 | 300 ~ 5000 人 | 百万 ~ 千万 | 非交互式日志量被严重低估；Event Hub TU 频繁限流 |
| 大型企业/集团 | 多租户 | > 5000 人 | 千万 ~ 亿级 | 多租户日志分别采集、隔离存储；服务主体数量庞大，基线建立困难 |

---

### 5.2 日志延迟

| 采集方案 | 各阶段延迟拆解 | 端到端延迟 | 是否满足准实时（5–15 分钟） |
|---|---|---|---|
| **方案一**（诊断设置→Event Hub） | Entra 内部批量打包（**~5 分钟，固定**）+ Event Hub 推送（秒级）+ 后台消费（秒级） | **6–15 分钟** | ✅ 满足（接近下限） |
| **方案二**（诊断设置→Storage→轮询） | 批量打包（~5 分钟）+ Storage 写入（秒级）+ 后台轮询间隔（5–15 分钟） | **10–30 分钟** | ⚠️ 边界情况 |
| **Graph API 轮询**（不推荐） | Graph API 实时可查，但轮询间隔 + 分页开销 | 取决于轮询频率，通常 5–30 分钟 | ⚠️ 理论可以，但成本高且有限流风险 |

**关键结论**：Entra 日志存在约 **5 分钟的批量打包窗口**——这是所有采集方案共同的延迟下限，无法通过任何配置压缩。即使走 Event Hub 实时路径，事件从发生到后台可消费的最快时间也约为 6 分钟。这比 Azure Activity Log（通常秒级到达 Event Hub）有更高的固定延迟。

**对研判引擎的要求**：同样需要以 `properties.createdDateTime`（SignInLogs）或 `properties.activityDateTime`（AuditLogs）为事件时间基准，配置 **≥ 15 分钟的乱序容忍窗口**（Late Data Tolerance）。不同类别的 Entra 日志（如 AuditLogs 和 SignInLogs）的批次到达时间可能不同，联合分析时需以事件时间对齐，不能以消息到达时间排序。

> **参考文档**：[Microsoft Entra activity log integration options](https://learn.microsoft.com/en-us/entra/identity/monitoring-health/concept-log-monitoring-integration-options-considerations)（原文："Events are batched into about 5-minute intervals"）

---

### 5.3 是否支持流式拉取

| 采集方案 | 数据流动模式 | 处理粒度 | 对关联分析的影响 |
|---|---|---|---|
| **方案一**（Event Hub） | **Push（5 分钟批次推送）** | 每批消息（含约 5 分钟内的所有事件） | 非真正流式，是"定时批次"模式；但比 Storage 轮询延迟低，是最接近实时的 Entra 采集路径 |
| **方案二**（Storage 轮询） | **定时 Pull（批量）** | 每小时一个文件 | 关联窗口至少需覆盖 1 小时 + 延迟缓冲；适合事后分析 |
| **Graph API**（补偿用途） | **主动 Pull（按需查询）** | 单条记录（支持 `$filter` 过滤） | 不适合持续采集；适合事后按条件查询特定时间段内的特定事件 |

**Entra 日志与 Azure Activity Log 的联合时序对齐**：

攻击链中的事件往往横跨 Entra 日志（登录）和 Azure Activity Log（ARM 操作）。两套日志分别通过各自的诊断设置路由，且批次推送的时间点可能不同（Entra 5 分钟批次 vs. Activity Log 秒级到达）。研判引擎在关联"Entra 登录事件 → ARM 操作"时，必须统一以各自记录内的事件时间字段为基准，而非以消息到达时间排序，并设置足够宽的关联窗口（建议 30 分钟）以覆盖两路延迟差异。

---

### 5.4 是否支持过滤

#### 层 1：诊断设置级别过滤（类别粒度）

Azure Diagnostic Settings 支持按**日志类别**选择是否路由：

| 过滤维度 | 能力 |
|---|---|
| **Entra 日志类别** | 可按 SignInLogs / AuditLogs / NonInteractiveUserSignInLogs 等分别开启或关闭；不需要的类别不会产生 Event Hub 消息费 |

**局限**：同样无法在诊断设置层面按 `userPrincipalName`、`appId`、`ipAddress` 等字段精确过滤，后台必须全量消费后自行解析和过滤。

#### 层 2：Graph API `$filter`（补偿查询）

Graph API 支持服务端 `$filter`，可按时间、用户、应用等字段过滤，适用于事后按条件查询特定日志，但**不适合作为持续采集的主路径**：

```http
GET https://graph.microsoft.com/v1.0/auditLogs/signIns?
    $filter=createdDateTime ge 2024-03-15T00:00:00Z
    and ipAddress eq '198.51.100.23'
    and status/errorCode ne 0
```

**非交互式日志的过滤策略**：由于非交互式登录量极大（5–10 倍于交互式），建议后台在消费 `NonInteractiveUserSignInLogs` 时，优先过滤掉来自已知可信服务的高频刷新（如 Teams、Exchange 的定期令牌刷新），聚焦于来自异常 IP、异常 AppId 或失败事件的记录，可大幅降低后续分析引擎的处理压力。

> **参考文档**：[Diagnostic settings in Azure Monitor](https://learn.microsoft.com/en-us/azure/azure-monitor/platform/diagnostic-settings)

### 5.5 限流策略

| 接口 | 限制 | 对后台的影响 |
|---|---|---|
| Event Hub 消费（AMQP） | 受 TU 限制：1 TU = 2MB/s 出流量；标准层支持 Auto-inflate | 非交互式日志高峰可能触发限流；建议开启 Auto-inflate 并设置 ≥ 4 TU 上限 |
| Event Hub 分区 | 同一 Consumer Group 内每分区只能被一个 Consumer 持有 | Worker 数量上限 = 分区数，建议分区数 ≥ 4 |
| Storage Blob 读取 | 无硬性 TPS 限制，超高并发可能触发 503 | 建议实现指数退避重试 |
| Graph API（`/auditLogs/signIns`） | 服务级别限流，高并发调用可能返回 429；每次最多 1000 条 | **仅用于补偿查询**，禁止作为主采集路径 |
| Entra Admin Center 内置下载 | 每次最多导出 100,000 条 | 仅用于人工调查，不适合程序化采集 |

### 5.6 日志时序一致性

#### 各方案时间字段语义

| 方案 | 正确时间基准（应使用） | 易混淆的"假时间" | 偏差量 |
|---|---|---|---|
| 方案一（Event Hub 消息，登录日志） | `properties.createdDateTime` | 消息外层 `time`（批次推送时间） | 约 5 分钟 |
| 方案一（Event Hub 消息，审计日志） | `properties.activityDateTime` | 消息外层 `time` | 约 5 分钟 |
| 方案二（Storage Blob，登录日志） | `properties.createdDateTime` | Blob 文件名中的小时时间戳 | 约 5–60 分钟 |
| Log Analytics（KQL 查询） | `TimeGenerated`（由 Log Analytics 从 `createdDateTime` 映射） | 通常准确，但首次摄入时可能有延迟 | 通常 < 5 分钟 |

> **Log Analytics 的特殊情况**：在 Log Analytics 中，`TimeGenerated` 字段是平台根据 `createdDateTime` 设置的，**不是日志摄入时间**，因此在 KQL 查询中使用 `TimeGenerated` 通常是正确的，与直接使用 `createdDateTime` 等效。

#### 跨日志类型的时序对齐

Entra 审计日志（AuditLogs）和登录日志（SignInLogs）的批次打包和推送时间点可能不同，导致两路日志在同一时间窗口内到达后台的顺序不确定。例如：攻击者登录（SignInLogs，T+0:00）→ 同意恶意应用（AuditLogs，T+0:02）这两个事件，可能后者（AuditLogs 批次）先到达后台，前者（SignInLogs 批次）后到达。研判引擎**必须以事件时间（而非消息到达时间）为基准对齐**，并设置合适的时间窗口宽度（建议 ≥ 15 分钟）。

### 5.7 多租户日志汇聚

Entra 日志的汇聚维度是**租户（Tenant）**，而非订阅。同一租户下所有订阅的用户和服务主体的身份行为，都汇聚到该租户唯一的一套 Entra 日志中。这与 Azure Activity Log（按订阅分别产生）不同，是一个天然的简化优势。

| 场景 | 汇聚方式 | 注意事项 |
|---|---|---|
| 单租户，多订阅 | **一套 Entra 诊断设置覆盖全部**，无需为每个订阅分别配置 | Entra 诊断设置在 Entra Admin Center 配置，与订阅无关 |
| 多租户（集团子公司各自独立租户） | 每个租户需要独立的 Entra 诊断设置和 Event Hub；后台维护多租户连接 | 不同租户的 Entra 日志互相独立，无法在单一诊断设置中跨租户汇聚 |
| 跨租户 B2B 访问 | 外部用户访问本租户资源时，本租户 Entra 日志中会有记录（`crossTenantAccessType: b2bCollaboration`）；但外部用户**在其原始租户的操作**对本租户不可见 | 需同时接入相关租户的 Entra 日志，才能完整还原跨租户的身份行为链 |

**给客户的接入建议**：识别客户是否存在合并租户/子公司独立租户的情况。集团客户可能有 3–10 个独立 Entra 租户，每个均需单独配置诊断设置和授权。B2B 协作场景下，攻击者可能在一个子公司租户中渗透后，以 Guest 身份访问总部租户的资源——两侧租户的 Entra 日志都需接入，才能看到完整的跨租户横向移动路径。

> **参考文档**：
> - [Microsoft Entra monitoring and health](https://learn.microsoft.com/en-us/entra/identity/monitoring-health/)
> - 跨租户访问：[Cross-tenant access overview](https://learn.microsoft.com/en-us/entra/external-id/cross-tenant-access-overview)

## 6. 安全厂商经验项总结

### 6.1 案例一：Storm-0558——伪造令牌访问政府机构邮箱（CISA + Microsoft，2023 年 7 月）

**事件背景**：

2023 年 6 月 16 日，一家美国联邦民事行政分支（FCEB）机构发现其 Microsoft 365 环境中有异常的邮件访问行为，并向微软报告。经微软调查确认，自 2023 年 5 月 15 日起，威胁行为者 Storm-0558（归因为中国国家背景）使用伪造的认证令牌，访问了约 25 个组织（包括多个政府机构）的用户邮件。

攻击者通过获取一枚微软 MSA（消费者账户）签名私钥，利用 Exchange Online 中的令牌验证缺陷，将消费者密钥伪造成企业 Azure AD 令牌，无感知地访问了 Exchange Online 邮件。

**Entra 日志的可见性分析**：

| 攻击阶段 | Entra 日志可见性 | 具体说明 |
|---|---|---|
| MSA 签名密钥泄露（2021 年） | ❌ 不可见 | 发生在微软内部，且"due to log retention policies, we don't have logs with specific evidence"——即便有日志，也因保留策略过期失效 |
| 攻击者使用伪造令牌访问 OWA | ⚠️ 部分可见 | Entra SignInLogs 中有登录记录，但令牌通过了验证逻辑，外观与正常登录完全相同；单靠 Entra 登录日志无法区分 |
| Exchange 邮件被批量访问 | ✅ **关键可见点**（M365 UAL） | **攻击是被 M365 UAL 中的 `MailItemsAccessed` 事件检测到的**，而非 Entra 登录日志。FCEB 机构发现"意外的 ClientAppID 和 AppID 在 M365 审计日志中产生了 MailItemsAccessed 事件"——这是整个案例的唯一有效检测点 |

**关键教训（针对 Entra 日志的视角）**：

1. **伪造令牌在 Entra SignInLogs 中外观合法**：因为令牌在 Exchange 的验证层通过了签名检查，Entra 日志中的登录记录显示 `status.errorCode: 0`（成功），`riskLevelAggregated: none`（无风险），Identity Protection 未触发任何风险事件。**仅凭 Entra 日志无法检测此类令牌伪造攻击**。

2. **`MailItemsAccessed` 是本案的决定性检测点，而非 Entra 日志**：CISA 建议报告中明确指出，检测此类攻击需要 M365 Purview Audit（Premium）的 `MailItemsAccessed` 功能，该功能历史上需要 E5 License（2023 年 9 月后对 E3 开放）。报告原文："The affected FCEB agency identified suspicious activity by leveraging enhanced logging—specifically of MailItemsAccessed events—and an established baseline of normal Outlook activity."

3. **日志保留不足导致溯源不完整**：微软在调查报告中承认，"由于日志保留策略，没有具体证据的日志来证明密钥外泄"。即便是平台提供商自身，也因日志保留期不足，无法完整还原超过两年的攻击路径。

4. **2023 年 9 月微软扩展了默认日志范围**：作为对 Storm-0558 事件的响应，微软宣布将包括 `MailItemsAccessed` 在内的 30 多类高级审计事件，从 E5 专属下放到 E3（Audit Standard）免费提供，同时将 Audit Standard 的默认保留期从 90 天延长到 180 天。这直接影响了安全厂商基于 M365 UAL 的告警检测能力边界评估。

> **参考来源**：
> - CISA + FBI 联合公告 AA23-193A：[Enhanced Monitoring to Detect APT Activity Targeting Outlook Online (CISA, Jul 2023)](https://www.cisa.gov/news-events/cybersecurity-advisories/aa23-193a)
> - Microsoft 技术分析：[Analysis of Storm-0558 techniques for unauthorized email access (Microsoft Security Blog, Jul 2023)](https://www.microsoft.com/en-us/security/blog/2023/07/14/analysis-of-storm-0558-techniques-for-unauthorized-email-access/)

### 6.2 案例二：OAuth 应用钓鱼 + 长期静默访问（Device Code Phishing，Proofpoint，2025 年）

**事件背景**：

Proofpoint 威胁情报团队在 2025 年记录了多组国家级行为者（包括俄罗斯 APT 组织）利用 OAuth Device Code 授权流程（Device Code Flow）持续实施账户接管的活跃攻击活动。攻击者向目标发送钓鱼邮件，诱导目标使用攻击者提供的 Device Code 授权一个应用程序，从而在不需要目标输入任何密码、不触发 MFA 的情况下，获得长期有效的 Access Token 和 Refresh Token。

**攻击链与 Entra 日志可见性**：

```
T+0  攻击者向目标发送钓鱼邮件，内含 device code 和伪装成合法服务的登录指引
T+1  目标在真实的微软登录页面（login.microsoftonline.com）输入 device code
     → Entra AuditLogs：activityDisplayName = "Consent to application"（若包含 scope）
     → Entra SignInLogs：记录了目标的交互式登录（成功，errorCode: 0）
T+2  攻击者后台收到 Access Token + Refresh Token（有效期通常各为 1 小时和 90 天）
     → 此步骤对 Entra 日志不可见（Token 传递发生在 OAuth 协议层）
T+3–T+90天  攻击者每小时静默刷新 Token，持续访问邮件、文件、Teams 消息
     → Entra NonInteractiveUserSignInLogs：每小时一条刷新记录，外观正常
     → M365 UAL（MailItemsAccessed 等）：记录数据访问行为
T+90天  Refresh Token 到期，攻击者若未更新则失去访问
```

**Entra 日志的检测机会分析**：

| 攻击阶段 | Entra 日志可见性 | 检测信号 |
|---|---|---|
| Device Code 授权（T+1） | ✅ **SignInLogs 有记录** | `clientAppUsed` 显示 Device Code Flow；登录地点与目标用户历史不符 |
| OAuth 同意（T+1，若有 scope） | ✅ **AuditLogs 有记录** | `activityDisplayName: Consent to application`；`additionalDetails.Permissions` 包含高权限 |
| Token 传递给攻击者（T+2） | ❌ **完全不可见** | Token 在 OAuth 协议层传递，不产生任何 Entra 日志记录 |
| 持续 Token 刷新（T+3~T+90天） | ⚠️ **NonInteractiveUserSignInLogs 有记录，但难以区分合法与恶意** | 刷新操作的 `appId` 是攻击者控制的应用；`ipAddress` 可能来自异常位置 |

**关键攻防洞察**：

- **Device Code Flow 绕过 MFA 的原理**：Device Code 授权流程允许用户在一个设备上授权，另一个设备获得访问权。攻击者利用此机制，使目标在自己的设备上完成了完整的 MFA 认证，而攻击者的设备获得了令牌——MFA 被正常执行了，但认证了错误的设备。Entra SignInLogs 中记录的登录显示 MFA 成功，**无法仅凭登录日志判断是否是钓鱼攻击**。
- **Refresh Token 的 90 天隐身期**：获得 Refresh Token 后，攻击者可在 90 天内每小时自动静默刷新一次 Access Token，产生的 NonInteractiveUserSignInLogs 记录与正常的 Outlook 客户端后台刷新完全相同。研判引擎需要建立**应用-用户-IP 三维行为基线**：该 AppId 是否是已知可信应用？该 IP 地址是否在用户的历史访问范围内？如果一个非白名单的 AppId 以用户身份从异地 IP 持续刷新令牌，是强检测信号。
- **撤销 Token 是终止攻击的有效手段**：一旦发现疑似攻击，可通过 Entra 吊销用户的所有刷新令牌（`Revoke-MgUserSignInSession`），立即中断攻击者的持续访问，而无需重置用户密码。这一操作本身会产生一条 AuditLogs 记录（`activityDisplayName: Revoke refresh tokens`），可用于确认响应操作的执行。

> **参考来源**：
> - Proofpoint — Device Code 钓鱼攻击（2025）：[Access Granted: Phishing with Device Code Authorization for Account Takeover (Proofpoint, Jan 2025)](https://www.proofpoint.com/us/blog/threat-insight/access-granted-phishing-device-code-authorization-account-takeover)
> - Microsoft — OAuth 应用滥用：[Threat actors misuse OAuth applications to automate financially driven attacks (Microsoft Security Blog, Dec 2023)](https://www.microsoft.com/en-us/security/blog/2023/12/12/threat-actors-misuse-oauth-applications-to-automate-financially-driven-attacks/)
> - Elastic Security Labs — Entra ID OAuth Phishing Detection（2025）：[Microsoft Entra ID OAuth Phishing and Detections (Elastic Security Labs, 2025)](https://www.elastic.co/security-labs/entra-id-oauth-phishing-detection)

### 6.3 核心经验项总结

**① Entra 日志 30 天默认保留期是最高优先级安全配置检查项**

这是接入 Entra 日志时的第一步必做动作，没有之一。Storm-0558 案例用最惨痛的方式证明了这一点——30 天后日志永久不可追溯。安全厂商应将"Entra 诊断设置是否已配置并指向持久存储"的检查，设置为账号接入 Checklist 的第 1 条，且接入后持续监控诊断设置是否被删除或修改。

**② License 等级决定了 Entra 日志的深度，而非仅仅保留期**

Free License 下，诊断设置无法导出登录日志；P1 可导出所有登录类别；P2 才能获取 Identity Protection 的风险信号（RiskyUsers、UserRiskEvents）。安全厂商在评估客户的检测能力边界时，必须先核查客户的 Entra License 等级，并明确告知各等级下可见与不可见的事件范围。

**③ 非交互式登录日志是量级放大器，必须纳入成本规划**

微软官方确认非交互式登录量为交互式的 5–10 倍。未将此因素纳入 Event Hub TU 和 Log Analytics 摄入成本规划，是最常见的导致超预算或日志丢失的工程错误。建议在接入前强制要求客户提供 1–2 天的日志样本量统计，再进行方案规划。

**④ Entra 日志单独不足以检测令牌伪造类攻击**

Storm-0558 案例证明，当攻击者持有合法（即使是伪造的）令牌时，Entra SignInLogs 中的登录记录看起来完全正常。检测这类攻击需要关联 M365 UAL 的数据面操作记录（如 `MailItemsAccessed`），以"异常 AppId 访问了邮件"而非"登录异常"作为检测依据。

**⑤ OAuth 应用的 AppId 白名单是检测 OAuth 钓鱼的核心机制**

无论是 OAuth 同意钓鱼（§4.5）还是 Device Code 钓鱼（案例二），攻击者最终都以一个非白名单的 AppId 获得令牌并持续访问资源。建议研判引擎建立租户内已知可信应用的 AppId 白名单，对任何非白名单 AppId 产生的登录记录（无论交互式还是非交互式）设置高优先级告警。

**⑥ 非交互式登录日志是攻击者长期隐蔽访问的"水下记录"**

攻击者在获得 Refresh Token 后，每小时一次的非交互式令牌刷新会产生大量外观正常的日志。安全厂商不应将非交互式登录日志视为低价值噪音而过度过滤，而应建立"应用-用户-IP"三维行为基线，对非白名单应用或异地 IP 的持续非交互式刷新设置聚合型告警（如：同一 AppId + 同一用户在 24 小时内从非历史 IP 段刷新 Token 超过 N 次）。

**⑦ 撤销刷新令牌是发现攻击后的第一响应动作**

确认攻击者已持有 Refresh Token 后，通过 Entra 管理员吊销目标用户的全部刷新令牌（`Revoke-MgUserSignInSession`）可立即终止其访问，比重置密码更快速且影响面可控。这一响应操作应写入 Playbook，且执行后需验证 AuditLogs 中出现了对应的 `Revoke refresh tokens` 记录。
