# CloudTrail 安全日志示例：第三方运维跨账号角色扮演入侵

## 关键字段对比与溯源逻辑

| 字段 | 账号A日志 | 账号B日志 |
|------|-----------|-----------|
| `userIdentity.type` | `IAMUser` | `AssumedRole` |
| `userIdentity.accountId` | `111111111111`（服务商） | `999999999999`（客户） |
| `userIdentity.accessKeyId` | `AKIA...`（长期凭证） | `ASIA...`（STS临时凭证） |
| `sessionContext.sessionIssuer.accountId` | — | `999999999999`，可回溯角色来源 |
| `sessionContext.attributes.mfaAuthenticated` | — | `false`，**未经 MFA 验证** |
| `sourceIPAddress` | `198.51.100.77` | `198.51.100.77`（**两侧一致，可关联**） |
| `roleSessionName` | `attacker-session` | 体现在 `principalId` 后缀中 |

## 检测规则建议

1. **跨账号 AssumeRole 基线告警**：账号 B 中出现 `userIdentity.type = AssumedRole` 且 `sessionIssuer.accountId ≠ recipientAccountId`，触发人工审核
2. **非常用 IP 跨账号跳转**：`sourceIPAddress` 不在服务商历史调用 IP 白名单内
3. **MFA 未认证的跨账号访问**：`mfaAuthenticated = false` 的跨账号 Session 应强制告警
4. **roleSessionName 异常**：Session 名称不符合企业命名规范（如随机字符串、攻击特征词）
5. **两侧日志联动关联**：以 `accessKeyId`（ASIA）+ `sourceIPAddress` 为主键，跨账号 A、B 两条 Trail 进行事件串联，还原完整攻击链
