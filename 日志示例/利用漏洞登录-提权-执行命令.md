# CloudTrail 日志示例（注释讲解版）：RCE 入侵 EC2 → 角色 a 提权 → AssumeRole 角色 b → SendCommand 横向

> **靶场环境**：
> - `ec2-ecs1`（`i-0a1b2c3d4e5f67890`）：绑定角色 a（`ec2-role-a`），对公网暴露 RCE 漏洞
> - `ec2-ecsA`（`i-0d3e4f5a6b789012`）：绑定角色 b 可管理，角色 a **原本无权访问**
> - 角色 a（`ec2-role-a`）：原始权限仅限管理 ecs1、ecs2；**不含 `sts:AssumeRole`**
> - 角色 b（`ec2-role-b`）：有权管理 ecsA、ecsB；Trust Policy 允许角色 a 扮演（配置错误）
>
> ⚠️ 以下所有日志为**伪 JSON 格式**，`//` 注释不符合 JSON 规范，仅供讲解使用，不可直接解析。

---

## 攻击时序总览

| 时间偏移 | 阶段 | 事件 | 使用凭据 |
|---|---|---|---|
| T-2h | **凭据颁发** | `AssumeRole`（由 `ec2.amazonaws.com` 代调） | —— |
| T+0:00 | 验证凭据 | `GetCallerIdentity` | 角色 a（`ASIA...ABCD`）|
| T+2:15 | 自我提权 | `AttachRolePolicy` | 角色 a（`ASIA...ABCD`）|
| T+5:33 | 角色跳转 | `AssumeRole` | 角色 a（`ASIA...ABCD`）|
| T+8:47 | 横向移动 | `SendCommand` | 角色 b（`ASIA...WXYZ`）|

## 阶段零：EC2 启动时颁发角色 a 临时凭据

**背景**：ecs1 启动并绑定 Instance Profile 时，AWS EC2 服务（`ec2.amazonaws.com`）自动代为调用 `sts:AssumeRole`，为实例颁发角色 a 的临时凭据。这是角色 a 临时凭据的「出生记录」，早于攻击者的任何操作存在于 CloudTrail 中。

> 攻击者后续通过 IMDS（`curl 169.254.169.254/...`）读取凭据的动作**不产生新的 CloudTrail 日志**——IMDS 读取只是从本地缓存取值，不触发新的 `AssumeRole`。因此本条日志是凭据来源可追溯的唯一证据。

```javascript
{
  "Records": [{

    // ── 事件元信息 ──────────────────────────────────────────────
    "eventVersion": "1.09",
    "eventTime": "2025-11-14T06:10:05Z",   // ⚠️ 早于攻击者验证凭据（T+0:00 = 08:21:44）约 2 小时
                                            //    对应阶段一日志中 sessionContext.attributes.creationDate 的值
                                            //    两者完全一致，可互相印证这是同一张凭据
    "eventSource": "sts.amazonaws.com",
    "eventName": "AssumeRole",             // EC2 服务代替实例调用 AssumeRole，颁发 Instance Profile 凭据
    "awsRegion": "us-east-1",
    "eventType": "AwsApiCall",
    "managementEvent": true,
    "readOnly": false,                     // 写操作，颁发了新的临时凭据

    // ── 调用者身份 ───────────────────────────────────────────────
    "userIdentity": {
      "type": "Service",                   // ⚠️ 固定值 "Service"；区别于用户/角色发起的调用
                                           //    表示本次调用由 AWS 内部服务代为发起，而非人工或应用程序
      "invokedBy": "ec2.amazonaws.com"     // ⚠️ EC2 服务专属字段；标识是 EC2 替实例自动完成的凭据颁发
                                           //    攻击者无法伪造此字段，它只在 AWS 服务内部调用时出现
    },

    // ── 网络信息 ─────────────────────────────────────────────────
    "sourceIPAddress": "ec2.amazonaws.com",
                                           // ⚠️ 非 IP 地址，而是服务域名
                                           //    表示请求由 AWS EC2 服务内部发起，没有客户端 IP
                                           //    与阶段一~四中攻击者的 45.142.212.18 形成鲜明对比
                                           //    凡是 sourceIPAddress 为服务域名的 AssumeRole，均是 AWS 服务代调
    "userAgent": "ec2.amazonaws.com",      // 与 sourceIPAddress 一致，均指向 EC2 服务

    // ── 请求参数 ─────────────────────────────────────────────────
    "requestParameters": {
      "roleArn": "arn:aws:iam::847293016482:role/ec2-role-a",
                                           // 被颁发凭据的目标角色，即绑定到 ecs1 的 Instance Profile 角色
      "roleSessionName": "i-0a1b2c3d4e5f67890",
                                           // ⚠️ Session Name 自动设为实例 ID
                                           //    与阶段一~三 userIdentity.arn 末段完全一致
                                           //    是将「凭据颁发记录」和「后续使用记录」关联起来的关键字段
      "durationSeconds": 3600              // 临时凭据有效期 1 小时；EC2 在到期前自动续期
    },

    // ── 响应（凭据内容）─────────────────────────────────────────
    "responseElements": {
      "credentials": {
        "accessKeyId": "ASIAIOSFODNN7ROLEABCD",
                                           // ⚠️ 颁发的临时 AK
                                           //    与阶段一~三 userIdentity.accessKeyId 完全一致
                                           //    是证明「本条日志颁发的凭据 = 攻击者后续使用的凭据」的直接证据
        "expiration": "Nov 14, 2025, 7:10:05 AM"
                                           // 凭据到期时间；EC2 通常在到期前 5 分钟自动续期
                                           // SecretAccessKey 和 SessionToken 不记录在 CloudTrail 中
      },
      "assumedRoleUser": {
        "assumedRoleId": "AROAIOSFODNN7ROLEAAAA:i-0a1b2c3d4e5f67890",
        "arn": "arn:aws:sts::847293016482:assumed-role/ec2-role-a/i-0a1b2c3d4e5f67890"
                                           // ⚠️ 与阶段一~三 userIdentity.arn 完全相同
                                           //    三阶段日志中的身份 ARN 正是从这里颁发出来的
      }
    },

    // ── 日志标识 ─────────────────────────────────────────────────
    "requestID": "z9y8x7w6-v5u4-3210-tsr9-qponmlkjih01",
    "eventID": "y8x7w6v5-u4t3-2109-srqp-onmlkjihgf02",
    "recipientAccountId": "847293016482"

  }]
}
```

**研判要点**：
- `userIdentity.type: "Service"` + `invokedBy: "ec2.amazonaws.com"` 是 Instance Profile 凭据颁发的唯一标识组合，正常情况下此类日志无需告警
- `responseElements.credentials.accessKeyId`（`ASIAIOSFODNN7ROLEABCD`）= 阶段一~三 `userIdentity.accessKeyId`，是跨阶段溯源的起点
- `eventTime`（`06:10:05Z`）= 阶段一 `sessionContext.attributes.creationDate`，两字段互相印证同一张凭据
- `requestParameters.roleSessionName: "i-0a1b2c3d4e5f67890"` 锁定了凭据归属的 EC2 实例，是定位受害机器的最早证据

## 阶段一：IMDS 窃取凭据，`GetCallerIdentity` 验证身份

**背景**：黑客通过 RCE 漏洞在 ecs1 获得 shell，从 `http://169.254.169.254/latest/meta-data/iam/security-credentials/ec2-role-a` 窃取临时凭据后，在自己的机器上调用此 API 验证凭据有效。

```javascript
{
  "Records": [{

    // ── 事件元信息 ──────────────────────────────────────────────
    "eventVersion": "1.09",              // CloudTrail 日志格式版本
    "eventTime": "2025-11-14T08:21:44Z", // 事件发生的 UTC 时间
    "eventSource": "sts.amazonaws.com",  // 调用的 AWS 服务
    "eventName": "GetCallerIdentity",    // 具体 API 名称；无需任何 IAM 权限即可调用，攻击者标准验证步骤
    "awsRegion": "us-east-1",            // 全局服务（STS）日志固定归集到 us-east-1
    "eventType": "AwsApiCall",           // 固定值，表示这是一次 API 调用记录
    "managementEvent": true,             // 管理事件，默认被 CloudTrail 记录
    "readOnly": true,                    // 只读操作，不修改任何资源

    // ── 调用者身份 ───────────────────────────────────────────────
    "userIdentity": {
      "type": "AssumedRole",             // ⚠️ 临时凭据标识；若是 IAM User 长期凭据则为 "IAMUser"
      "principalId": "AROAIOSFODNN7ROLEAAAA:i-0a1b2c3d4e5f67890",
                                         // 冒号前：Role 的唯一 ID；冒号后：Session Name，Instance Profile 自动以实例 ID 命名
      "arn": "arn:aws:sts::847293016482:assumed-role/ec2-role-a/i-0a1b2c3d4e5f67890",
                                         // 最后一段 i-0a1b2c3d4e5f67890 = Session Name = 受害实例 ID，可据此定位是哪台 EC2 被攻陷
      "accountId": "847293016482",       // 被攻击的 AWS 账号 ID
      "accessKeyId": "ASIAIOSFODNN7ROLEABCD",
                                         // ⚠️ 前缀 ASIA = STS 临时凭据；AKIA = IAM User 长期凭据；两者可从前缀直接区分
      "sessionContext": {
        "sessionIssuer": {
          "type": "Role",                // 凭据由 Role 签发（而非 User）
          "principalId": "AROAIOSFODNN7ROLEAAAA", // Role 自身的唯一 ID
          "arn": "arn:aws:iam::847293016482:role/ec2-role-a",
                                         // ⚠️ 溯源关键字段：签发这张临时凭据的原始 Role ARN
          "accountId": "847293016482",
          "userName": "ec2-role-a"       // 原始 Role 名称
        },
        "ec2RoleDelivery": "1.0",        // ⚠️ EC2 Instance Profile 专属字段；凭据通过 IMDS 下发才会出现此字段
                                         //    AssumeRole / OIDC 等方式获得的凭据不含此字段，有无可判断凭据来源
        "attributes": {
          "creationDate": "2025-11-14T06:10:05Z",
                                         // 凭据颁发时间；与本次调用时间差约 2h11m，说明凭据并非刚刚颁发
                                         // EC2 Instance Profile 凭据每小时自动轮换
          "mfaAuthenticated": "false"    // 临时凭据不涉及 MFA，固定为 false
        }
      }
    },

    // ── 网络信息 ─────────────────────────────────────────────────
    "sourceIPAddress": "45.142.212.18",  // ⚠️ 核心异常：境外商业 VPS；EC2 实例正常调用来源应为实例内网 IP（10.x.x.x）
                                         //    ec2RoleDelivery="1.0" + 非实例 IP = IMDS 凭据被带出实例在外部使用
    "userAgent": "aws-cli/2.17.23 Python/3.12.2 Linux/6.1.0 botocore/2.4.5",
                                         // 辅助异常：EC2 上运行的应用通常使用 SDK（如 aws-sdk-java），裸 aws-cli 不符合正常应用行为
                                         // userAgent 可伪造，仅作辅助判断

    // ── 请求与响应 ───────────────────────────────────────────────
    "requestParameters": null,           // GetCallerIdentity 无入参，null 为正常现象
    "responseElements": {
      "arn": "arn:aws:sts::847293016482:assumed-role/ec2-role-a/i-0a1b2c3d4e5f67890",
      "userId": "AROAIOSFODNN7ROLEAAAA:i-0a1b2c3d4e5f67890",
      "account": "847293016482"          // 攻击者通过响应确认了账号 ID，为后续枚举提供基础
    },

    // ── 日志标识 ─────────────────────────────────────────────────
    "requestID": "a1b2c3d4-e5f6-7890-abcd-ef1234567801", // 单次请求的唯一 ID
    "eventID": "b2c3d4e5-f6a7-8901-bcde-f12345678902",   // CloudTrail 事件的唯一 ID，用于去重
    "recipientAccountId": "847293016482" // 接收请求的账号 ID；与 accountId 相同说明是同账号操作（非跨账号）

  }]
}
```

## 阶段二：角色 a 给自身附加 AssumeRole 权限

**背景**：角色 a 原本不含 `sts:AssumeRole`，但其策略允许 `iam:AttachRolePolicy`（针对自身）。攻击者利用此权限为角色 a 附加包含 `sts:AssumeRole` 的策略，为跳转角色 b 做准备。

```javascript
{
  "Records": [{

    // ── 事件元信息 ──────────────────────────────────────────────
    "eventVersion": "1.09",
    "eventTime": "2025-11-14T08:23:59Z",  // 距阶段一约 2 分 15 秒，自动化工具的典型速度
    "eventSource": "iam.amazonaws.com",   // IAM 服务；IAM 是全局服务，日志固定归集到 us-east-1
    "eventName": "AttachRolePolicy",      // ⚠️ 高危写操作：为角色附加策略
    "awsRegion": "us-east-1",
    "eventType": "AwsApiCall",
    "managementEvent": true,
    "readOnly": false,                    // ⚠️ 写操作；修改了 IAM 资源配置

    // ── 调用者身份（与阶段一完全相同，同一组凭据）──────────────
    "userIdentity": {
      "type": "AssumedRole",
      "principalId": "AROAIOSFODNN7ROLEAAAA:i-0a1b2c3d4e5f67890",
      "arn": "arn:aws:sts::847293016482:assumed-role/ec2-role-a/i-0a1b2c3d4e5f67890",
      "accountId": "847293016482",
      "accessKeyId": "ASIAIOSFODNN7ROLEABCD", // 与阶段一相同，确认是同一组窃取凭据
      "sessionContext": {
        "sessionIssuer": {
          "type": "Role",
          "principalId": "AROAIOSFODNN7ROLEAAAA",
          "arn": "arn:aws:iam::847293016482:role/ec2-role-a",
          "userName": "ec2-role-a"
        },
        "ec2RoleDelivery": "1.0",             // 仍为 IMDS 下发的凭据，与阶段一一致
        "attributes": {
          "creationDate": "2025-11-14T06:10:05Z",
          "mfaAuthenticated": "false"
        }
      }
    },

    // ── 网络信息 ─────────────────────────────────────────────────
    "sourceIPAddress": "45.142.212.18",   // 与阶段一相同，跨阶段关联的核心依据

    "requestParameters": {
      "roleName": "ec2-role-a",           // ⚠️ 被附加策略的角色名
                                          //    与 userIdentity.sessionContext.sessionIssuer.userName 相同

      "policyArn": "arn:aws:iam::847293016482:policy/AllowAssumeRoleB"
                                          // ⚠️ 被附加的策略 ARN；此为账号内客户托管策略（非 AWS 托管）
                                          //    策略内容：允许 sts:AssumeRole，目标为 ec2-role-b
                                          //    附加后角色 a 的当前 Session 立即生效，无需重新颁发凭据
    },
    "responseElements": null,             // IAM 写操作成功时 responseElements 为 null，属正常现象

    // ── 日志标识 ─────────────────────────────────────────────────
    "requestID": "c3d4e5f6-a7b8-9012-cdef-012345678903",
    "eventID": "d4e5f6a7-b8c9-0123-def0-123456789004",
    "recipientAccountId": "847293016482"

  }]
}
```

## 阶段三：角色 a 扮演角色 b，获取新临时凭据

**背景**：附加策略后，攻击者立即调用 `AssumeRole` 扮演角色 b，拿到一组全新的临时凭据。**此后所有操作切换到角色 b 的凭据**，`accessKeyId` 在本阶段响应中完成交接。

```javascript
{
  "Records": [{

    // ── 事件元信息 ──────────────────────────────────────────────
    "eventVersion": "1.09",
    "eventTime": "2025-11-14T08:27:17Z",  // 距阶段二约 3 分 18 秒
    "eventSource": "sts.amazonaws.com",
    "eventName": "AssumeRole",            // ⚠️ 角色跳转核心事件；调用者是角色 a，目标是角色 b
    "awsRegion": "us-east-1",

    "managementEvent": true,
    "readOnly": false,                    // 写操作，颁发了新的临时凭据
```

## 阶段四：角色 b 对 ecsA 执行 SendCommand 反弹 Shell

**背景**：攻击者换用角色 b 的凭据，向 ecsA 下发 SSM 命令执行反弹 shell。命令内容在 CloudTrail 中被强制隐藏。

```javascript
{
  "Records": [{

    // ── 事件元信息 ──────────────────────────────────────────────
    "eventVersion": "1.09",
    "eventTime": "2025-11-14T08:30:31Z",  // 距阶段三约 3 分 14 秒
    "eventSource": "ssm.amazonaws.com",   // SSM（Systems Manager）服务
    "eventName": "SendCommand",           // ⚠️ 向目标实例下发命令；是横向移动的核心事件
    "awsRegion": "us-east-1",
    "eventType": "AwsApiCall",
    "managementEvent": true,
    "readOnly": false,                    // 写操作，在目标实例上执行了命令

    // ── 调用者身份（已切换为角色 b 的凭据）──────────────────────
    "userIdentity": {
      "type": "AssumedRole",
```

## 日志之间如何找到对方

本攻击链涉及四条 CloudTrail 日志和一组 CloudWatch 日志，分属不同数据源、使用不同字段结构。以下说明每对日志之间的具体关联方式。

---

### 关联一：阶段一 → 阶段二 → 阶段三（同一组凭据的连续操作）

**关联字段**：`userIdentity.accessKeyId`

三个阶段使用的是同一组角色 a 临时凭据，`accessKeyId` 完全相同（`ASIAIOSFODNN7ROLEABCD`）。

```
阶段一 userIdentity.accessKeyId = "ASIAIOSFODNN7ROLEABCD"
阶段二 userIdentity.accessKeyId = "ASIAIOSFODNN7ROLEABCD"  ← 相同，同一组凭据
阶段三 userIdentity.accessKeyId = "ASIAIOSFODNN7ROLEABCD"  ← 相同，同一组凭据
```

**辅助字段**：`sourceIPAddress`（三阶段均为 `45.142.212.18`）、`userIdentity.arn`（均含 `ec2-role-a/i-0a1b2c3d4e5f67890`）

**查询方式**（CloudTrail Lake）：
```sql
SELECT eventTime, eventName, userIdentity.accessKeyId, sourceIPAddress
FROM cloudtrail_logs
WHERE userIdentity.accessKeyId = 'ASIAIOSFODNN7ROLEABCD'
ORDER BY eventTime ASC
```

---

### 关联二：阶段三 → 阶段四（凭据交接，跨角色关联）

**这是整条链最难关联的一跳**：阶段四的执行者是角色 b，`accessKeyId` 已切换为 `WXYZ`，与前三阶段的 `ABCD` 不同，无法直接用同一 AK 关联。

**关联字段**：阶段三的 `responseElements.credentials.accessKeyId`

阶段三 `AssumeRole` 的响应中，`responseElements` 明文记录了新颁发给角色 b 的 `accessKeyId`，其值恰好等于阶段四 `userIdentity.accessKeyId`。

```
阶段三 responseElements.credentials.accessKeyId = "ASIAIOSFODNN7ROLEWXYZ"
                                                            ↕ 相等
阶段四 userIdentity.accessKeyId                 = "ASIAIOSFODNN7ROLEWXYZ"
```

**查询方式**：先从阶段四的 AK 反查颁发它的 `AssumeRole` 事件：
```sql
-- 第一步：已知阶段四的 accessKeyId，找到颁发它的 AssumeRole 事件
SELECT eventTime, eventName,
       userIdentity.arn            AS caller,
       requestParameters.roleArn   AS target_role,
       responseElements            AS new_credentials
FROM cloudtrail_logs
WHERE eventName = 'AssumeRole'
  AND json_extract_scalar(responseElements, '$.credentials.accessKeyId')
      = 'ASIAIOSFODNN7ROLEWXYZ'   -- 从阶段四的 accessKeyId 反向查找
```

**辅助字段**：`userIdentity.arn` 中的 Session Name——阶段三响应里的 `assumedRoleUser.arn` 末段为 `pivot-to-role-b`，阶段四 `userIdentity.arn` 末段同样为 `pivot-to-role-b`，两者字符串完全一致。

```
阶段三 responseElements.assumedRoleUser.arn
  = "arn:aws:sts::847293016482:assumed-role/ec2-role-b/pivot-to-role-b"
                                                                ↕ 末段相同
阶段四 userIdentity.arn
  = "arn:aws:sts::847293016482:assumed-role/ec2-role-b/pivot-to-role-b"
```

### 关联三：阶段四（CloudTrail）→（CloudWatch）

**关联字段**：`commandId`

阶段四 `SendCommand` 的 `responseElements.command.commandId` 与 CloudWatch Log Stream 的路径前缀是同一个值，两者通过这个 ID 直接绑定。

```
CloudTrail 阶段四：
  responseElements.command.commandId = "7a8b9c0d-1e2f-3456-789a-bcdef0123456"
                                                    ↕ 完全相同
CloudWatch Log Stream 路径：
  7a8b9c0d-1e2f-3456-789a-bcdef0123456/i-0d3e4f5a6b789012/aws-runShellScript/stdout
```

**定位步骤**：
1. 从 CloudTrail `SendCommand` 事件取出 `commandId`
2. 进入 CloudWatch Logs，Log Group `/aws/ssm/AWS-RunShellScript`
3. 按前缀 `7a8b9c0d-1e2f-3456-789a-bcdef0123456` 过滤 Log Stream
4. 打开 `stdout` 流，第二条 `message` 即为命令明文

### 各关联字段的可靠性对比

| 关联字段 | 可伪造 | 关联强度 | 说明 |
|---|---|---|---|
| `accessKeyId` | 否 | ⭐⭐⭐⭐⭐ | AWS 颁发的唯一标识，不可伪造 |
| `responseElements.credentials.accessKeyId` → 下一阶段 `userIdentity.accessKeyId` | 否 | ⭐⭐⭐⭐⭐ | 凭据交接的唯一证明，字段值由 AWS 生成 |
| `commandId` → CW Log Stream 前缀 | 否 | ⭐⭐⭐⭐⭐ | AWS 生成，两侧强绑定 |
| `userIdentity.arn`（含 Session Name）| **是**（Session Name 由调用方填写）| ⭐⭐⭐⭐ | 辅助关联；`pivot-to-role-b` 可作为 IoC 但技术上可填任意值 |
| `sourceIPAddress` | **是**（可用代理）| ⭐⭐⭐ | 强关联但可规避；攻击者若换 IP 则断链 |
| `userAgent` | **是**（完全可伪造）| ⭐ | 仅辅助参考，不可作为关联主依据 |
